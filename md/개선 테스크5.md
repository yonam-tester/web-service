# 개선 테스크 5: MinIO 복구 파이프라인, 프론트엔드 API 키 제어 및 UI/텍스트 고도화

본 문서는 연암 테스터 애플리케이션의 완성도를 기성 서비스 수준으로 끌어올리기 위한 **MinIO 기반 DB 자동 복구, API 키 보안 제어 단일화 및 피드백, 그리고 결과 화면 UI/텍스트의 정밀 매핑 파이프라인** 구현 계획을 수립한 명세서입니다.

---

## 🛠️ 개선사항 1: MinIO(S3) 파일 기반 대시보드 동기화 복구 파이프라인 (Task 5.1)

### 1. 배경 및 목적
- 백엔드 재시작 시 H2 인메모리 특성으로 인해 DB 레코드가 초기화되더라도, MinIO(S3)에는 물리 기획서 파일이 그대로 영구 보존되어 있습니다.
- 이를 활용하여 백엔드가 재기동되거나 사용자가 대시보드를 로드할 때 MinIO 버킷을 스캔하여 유실된 `Project` 및 `UploadedFile` 메타데이터를 RDB로 복구 적재해 주는 자동 복구 동기화 파이프라인을 구축합니다.

### 2. 세부 구현 계획
- **스캔 및 파싱 엔진 구현 (`S3SyncService.java` 신설):**
  - 백엔드 로드 시 `ApplicationReadyEvent` 리스너를 트리거하거나, 대시보드 진입 API (`GET /api/projects`) 호출 시 비동기로 작동합니다.
  - `s3Client.listObjectsV2`를 호출하여 `yeonam-documents` 버킷 내의 모든 객체를 스캔합니다.
  - S3 키 경로(`projects/{projectId}/{fileId}_{fileName}`)를 정규식으로 파싱하여 `projectId`, `fileId`, `fileName`을 추출합니다.
- **RDB 레코드 복구 로직:**
  - 파싱된 `projectId`에 해당하는 프로젝트가 RDB에 없을 경우, 기본값(프로젝트명: "복구된 프로젝트_" + ID, 설명: "MinIO S3 동기화에 의해 복구된 프로젝트입니다.")으로 `Project` 테이블에 선제 인서트합니다.
  - 이어서 `fileId`에 해당하는 파일이 `UploadedFile` 테이블에 없을 경우, `status = DONE`, `fileType = REQUIREMENT_SPEC` 정보와 함께 레코드를 생성하여 프로젝트와 매핑합니다.
- **안전장치:**
  - 기존에 이미 DB에 존재하는 레코드에 대해서는 중복 인서트를 방지하도록 `existsById` 검증을 필수로 수행합니다.

## 🛠️ 개선사항 2: 프론트엔드 API 키 제어 및 LLM/RAG 연동 & 오류 피드백 (Task 5.2)

### 1. 배경 및 목적
- 보안 규정에 맞춰 API 키는 오직 프론트엔드(Client LocalStorage)에서만 관리하도록 제한합니다.
- 분석 실행 시 이 API 키를 백엔드 API의 바디에 실어 RAG/LLM 서버로 전송하고, 만약 API 키가 누락되었거나 만료/오류가 났을 때 "유효하지 않은 API 키" 피드백이 사용자 화면에 정확히 에러 메시지로 표출되도록 오류 전파 통로를 일원화합니다.

### 2. 세부 구현 계획
- **프론트엔드 API 키 단일 전송:**
  - `DocumentUploadPage.tsx`에서 분석 시작 버튼을 클릭할 때 로컬 스토리지에서 수집한 `yeonam_llm_api_key`를 `AnalysisCreateRequest` DTO의 `llmApiKey` 필드에 바인딩하여 백엔드로 보냅니다.
- **백엔드 토스 파이프라인:**
  - `AnalysisService.java` -> `TriggerRequest` DTO -> FastAPI `POST /api/analysis/trigger` 로 API 키를 토스합니다.
- **AI/RAG 서버 검증 및 실패 콜백 전파:**
  - `rag_server` 및 `llm_server`는 수신한 `llmApiKey`로 LLM API를 호출하며, API Key 인증 오류(AuthenticationError, 401) 발생 시 즉시 예외를 캐치합니다.
  - 캐치한 예외를 바탕으로 백엔드 콜백 API를 즉시 호출하여 `status = FAILED`, `errorMessage = "API 키 유효성 검증 실패: 유효하지 않은 API 키이거나 만료되었습니다. 키 설정을 재점검해 주세요."`를 전송합니다.
  - 백엔드 `CallbackService.java`는 이 콜백을 받아 `AnalysisJob`의 상태를 `FAILED`로 변경하고, `summary` 컬럼에 이 에러 메시지를 영속화합니다.
- **프론트엔드 UI 오류 피드백:**
  - 폴링 컴포넌트는 `FAILED` 상태 감지 시, `summary`에 저장된 오류 사유를 가져와 분석 진행 팝업 창 내에 강렬한 빨간색 경고창으로 띄워 학생이 즉시 피드백을 인지하고 조치하도록 개선합니다.

## 🛠️ 개선사항 3: 분석 명세서 결과 화면 UI 및 텍스트 구조 개선 (Task 5.3)

### 1. 배경 및 목적
- 기존 테스트 케이스 카드 배치가 세로로만 나열되어 스크롤이 지나치게 길어지는 사용성 문제를 해결합니다.
- 또한, RAG 분석을 거쳐 도출된 실제 '기획서 누락 분석(Missing Items)' 텍스트가 하드코딩 더미 텍스트 대신 RAG 서버의 실제 분석 결과물과 정밀 매핑되도록 백엔드-프론트엔드 API를 연동합니다.

### 2. 세부 구현 계획
- **가로 2열 카드 그리드 레이아웃 개선:**
  - `AnalysisResultPage.tsx`의 테스트 케이스 목록 렌더링 영역을 `flex flex-col` 구조에서 `grid grid-cols-1 md:grid-cols-2 gap-6` 구조로 확장 개선합니다.
  - 카드 내에서 스크롤을 방해하던 장문 TDD 가이드 등의 세부 아코디언 컴포넌트를 카드 크기에 맞춰 컴팩트하게 리사이징합니다.
- **실제 누락 기획 명세서 텍스트 매핑 파이프라인:**
  - RAG 서버가 요구사항 분석 시 도출하여 전송하는 `missing_items` 목록을 백엔드 RDB의 `AnalysisJob` 엔티티 내에 새로운 컬렉션(또는 `TEXT` 포맷의 세미콜론 구분 문자열 등)으로 저장할 수 있도록 컬럼을 확장합니다.
  - `AnalysisResultResponse`에 이 실제 적재된 `missingItems` 리스트를 바인딩하여 프론트엔드로 전달합니다.
  - 프론트엔드 결과 페이지의 우측 '기획 명세서 누락 분석' 가이드 뷰포트 내에 더미 텍스트 대신 이 실제 수신된 데이터 리스트가 아이콘과 함께 표출되도록 컴포넌트 데이터 바인딩을 완료합니다.
