# 연암 테스터 (QA/TDD 검증 보조 플랫폼) 시스템 아키텍처 및 실행 흐름

## 1. 아키텍처 구성 요소 및 직관적 역할 정의

### 📱 Client Tier: React Frontend

사용자와 시스템이 소통하는 웹 대시보드로, 복잡한 분석 과정을 직관적으로 제어하고 시각화합니다.

* **입력 및 제어:** 프로젝트 메타 정보(GitHub URL 등) 입력, 파일 업로드 시 사전 유효성 검사(용량 및 확장자 차단), 분석을 위한 QA 관점(프론트/백엔드/보안 등) 선택 및 명시적 분석 시작 요청 기능.
* **실시간 모니터링:** 전체 파이프라인 단계별 진행 상태(스피너 등) 및 통계를 시각적 피드백으로 제공.
* **결과 렌더링 (카드 UI):** 분석 요약, 테스트 케이스 상세, 위험 요소 태그, 우선순위 배지 등을 카드 형태로 표시하며, 긴 근거 문장은 말줄임표와 아코디언(접기/펼치기) UI로 깔끔하게 제공.
* **보고서 제어:** 브라우저 내에서 최종 보고서를 미리보기하고 다운로드 요청.

### ⚙️ Application Tier: Spring Boot API Server

비즈니스 로직의 중심이자 파이프라인의 컨트롤 타워입니다.

* **API 및 작업 관리:** 프론트엔드의 요청(프로젝트 생성, 파일 업로드, 분석 상태 조회, 보고서 다운로드)을 처리하는 REST API 제공.
* **무결성 및 통신 관리:** 업로드된 파일의 백엔드 무결성 재검증, S3 저장소 연동, RAG 서버로의 비동기 분석 작업 지시 및 결과 반환.
* **메타데이터 기록:** 프로젝트 단위 및 분석 작업 단위(AnalysisJob) 메타데이터, 파일 상태를 관계형 DB(H2)에 기록 및 추적.
* **비동기 웹훅 대기:** LLM 호출을 직접 수행하지 않고, AI 서버로 지시 후 내부 콜백 엔드포인트(`/api/internal/analysis/{analysisId}/callback`)를 통해 결과를 수신.

### 🧠 Analysis Tier: AI 분석 계층 (LLM Server / RAG Server 교체형 구조)

본 계층은 스프링부트와 완전히 독립되어 구동되며, 개발 초기 단계(MVP)와 최종 고도화 단계에 따라 두 가지 서버 포맷으로 갈아 끼울 수 있습니다 (이전 대화 내용 반영). 두 서버는 **완벽히 동일한 API 규격
(웹훅 콜백 등)**을 공유하여 스프링부트 측의 코드 수정을 요구하지 않습니다.

**Type A. LLM Server (초기 MVP용 핵심 기능 공장)**
복잡한 Vector DB 없이 순수 텍스트 처리와 LLM 연동에 집중하는 경량 서버입니다.
* **1단계 (입고):** 업로드된 문서를 `Apache Tika` 등을 통해 텍스트만 추출하며, 빠른 처리를 위해 메모리(RAM)에 임시 적재합니다 (상태: `UPLOADED`).
* **2단계 (작업 지시):** Spring Boot의 비동기 호출을 받아 대기열에 등록합니다 (상태: `QUEUED`).
* **3단계 (제조 과정):** 텍스트를 파싱하고 `MockLlmClient` 또는 실제 외부 LLM API(`LiteLLM` 활용)를 호출하여 JSON 형태의 테스트 케이스 생성을 요청합니다 (상태: `PARSING` / `GENERATING`).
* **4단계 (출고):** 조립된 최종 JSON 결과물을 Spring Boot의 콜백 API로 쏘아주어(Webhook) 작업을 완료합니다 (상태: `COMPLETED` / `FAILED`).

**Type B. RAG Server (최종 고도화 플랫폼)**
LLM Server의 인터페이스를 유지하면서, 내부에 고급 검색 증강 로직을 추가한 형태입니다.
* **문서 전처리 및 청킹:** `DocumentParser`를 통해 복합 문서의 레이아웃을 보존하여 텍스트를 추출하고 논리적 크기로 분할(Chunking).
* **요구사항 독립 추출:** `RequirementExtractor`로 검증 가능한 요구사항 단위를 분리하여 식별자 부여.
* **복합 검색 (Cross-Search):** `Retriever`를 통해 추출 요구사항과 사전에 구축된 QA 지식베이스를 교차 검색하여 테스트 설계 기준/위험 사례 도출.
* **프롬프트 조립 및 파싱:** 도출된 근거(최대 3개)를 사용자 지시문과 병합(`PromptBuilder`)하고 외부 LLM을 호출하여 결과를 구조화 파싱(`ResultFormatter`).

### 💾 Storage Tier: 통합 데이터 저장소 계층

데이터의 성격에 맞춰 분리된 3가지 최적화 저장소를 활용합니다.

* **파일 저장소 (S3):** 원본 업로드 문서(PDF, MD 등)와 최종 산출된 물리적 검증 보고서 보관.
* **관계형 DB (H2 DB):** 프로젝트 생성 메타데이터, 추출 요구사항, 테스트 케이스, 그리고 1:N(테스트:다중 근거) 추적성 매핑 데이터 트랜잭션 기록.
* **Vector DB (FAISS / Bedrock Knowledge Base):** 청킹된 문서 조각 및 QA 도메인 지식의 다차원 임베딩 벡터값 저장 및 Top-K 유사도 고속 탐색. (초기 MVP 비용 절감을 위해 로컬 FAISS로 구축하며, 향후 클라우드 네이티브 환경 및 S3 연동 고도화 시 AWS Bedrock Knowledge Base를 원활하게 사용할 수 있도록 유연하게 설계합니다.)

#### 2. 아키텍처 간 실행 흐름 (연결 방식)

각 구성요소가 하나의 자동화된 파이프라인으로 연결되는 흐름입니다.

1.  **초기 설정 및 업로드 (React ↔ Spring Boot ↔ S3 / H2)**
    *   사용자가 React에서 프로젝트 정보를 입력하고 문서를 업로드합니다.
    *   Spring Boot는 무결성 검증 후 원본을 S3에 저장하고, 분석 대기 상태를 H2 DB에 기록합니다.
2.  **분석 트리거 및 지시 (React → Spring Boot → Analysis Tier)**
    *   '분석 시작' 요청 시 Spring Boot가 선택된 Analysis Tier(LLM 또는 RAG 서버)로 비동기 분석 작업과 파라미터를 전달합니다.
3.  **데이터 제조 및 AI 통신 (Analysis Tier ↔ Vector DB ↔ LLM API)**
    *   (LLM 서버의 경우) 텍스트를 정제하여 LLM과 즉시 통신합니다.
    *   (RAG 서버의 경우) Vector DB에 문서를 인덱싱하고, QA 지식베이스를 탐색하여 관련 근거를 찾은 뒤 LLM 프롬프트를 조립하여 전송합니다.
4.  **콜백 수신 및 매핑 저장 (Analysis Tier → Spring Boot ↔ H2 DB)**
    *   분석 서버가 LLM 응답을 객체화하여 Spring Boot의 콜백(Webhook) 엔드포인트로 반환합니다.
    *   Spring Boot는 전달받은 테스트 케이스, 다중 근거, 우선순위 데이터를 H2 DB에 트랜잭션 단위로 안전하게 매핑하여 저장합니다.
5.  **렌더링 및 파일 반출 (H2 DB ↔ Spring Boot ↔ React ↔ S3)**
    *   React는 저장된 DB 정보를 바탕으로 직관적인 대시보드 카드 UI를 렌더링합니다.
    *   보고서 산출 시 종합 데이터를 Markdown/PDF로 변환하여 S3에 저장한 후 다운로드 형태로 반출합니다.

#### 3. 기술 아키텍처 및 구현 방향

1.  **프론트엔드 (Client Tier) 및 백엔드 (Application Tier)**
*  프론트엔드: React를 사용하여 파일 업로드, 진행 상태 모니터링, 카드형 결과 조회 및 Markdown/PDF 보고서 미리보기 UI를 구현합니다. 브라우저 내 보고서 렌더링을 위해 마크다운 리더기 컴포넌트를 탑재합니다.
*  AI 어시스턴트를 활용하여 개발하되, 최종 산출물은 일반적인 React 애플리케이션입니다.
*  백엔드: Spring Boot가 파이프라인의 컨트롤 타워 역할을 수행합니다. 프로젝트 메타데이터, 업로드 파일 상태, 테스트 케이스 결과 매핑 등 전반적인 트랜잭션 관리를 위해 관계형 데이터베이스인 H2 DB를 사용합니다.
*  애플리케이션 내에 MCP 서버 모듈은 포함하지 않으며, 프론트엔드와는 표준 HTTP/REST API를 통해 통신합니다.
2.  **Spring Boot ↔ RAG Server 통신 방식 (가장 유연한 연결)**
*  RAG 서버가 향후 어떤 언어(Python 등)나 프레임워크로 구현되더라도 쉽게 연결할 수 있도록, 가장 보편적이고 결합도가 낮은 HTTP 기반 REST API + 비동기 콜백(Webhook) 또는 폴링(Polling) 방식을 기본으로 채택합니다.
*  Spring Boot가 RAG 서버로 분석에 필요한 데이터(파일 위치, 요구사항, 파라미터 등)를 POST 요청으로 전달하고, RAG 서버는 분석이 완료되면 Spring Boot의 특정 엔드포인트(Webhook)로 결과를 쏴주거나, Spring Boot가 주기적으로 상태를 묻는(Polling) 형태가 가장 적합합니다.
3.  **로컬 스토리지 (S3 호환)**
*  초기 개발 및 테스트 환경에서는 AWS S3와 API가 100% 호환되는 로컬 스토리지인 MinIO 또는 LocalStack을 우선 적용하겠습니다.
*  Spring Boot 내부에서는 AWS S3 SDK를 그대로 사용하며, 엔드포인트(Endpoint) URL만 로컬 MinIO로 바라보게 설정합니다. 추후 실제 운영 단계에서는 코드 변경 없이 환경 변수(인증키, S3 버킷 주소)만 교체하면 즉시 AWS S3로 전환되어 온전하게 동작합니다.
4.  **비결합형(Decoupled) 통신 설계**
*  Spring Boot와 AI 분석 서버(LLM/RAG)는 가장 범용적인 HTTP 기반 REST API와 비동기 콜백(Webhook) 방식을 사용합니다.
*  이를 통해 AI 서버가 Python 기반의 FastAPI 등으로 작성되거나 향후 엔진이 변경되더라도 백엔드 코드는 수정 없이 URL 환경변수만 교체하면 즉시 호환됩니다.
*  AI 분석 서버는 문서 파싱을 위해 Apache Tika 라이브러리를 활용하며, 외부 상용 LLM 통신을 위해 LiteLLM 게이트웨이를 적용하고, 초기 개발 및 테스트 비용 절감을 위한 MockLlmClient 연동 모드를 지원합니다.
5.  **로컬 및 클라우드 호환성**
*  스토리지 인프라는 AWS S3 SDK와 100% 호환되는 로컬 스토리지(MinIO)를 우선 적용하며, Vector DB 역시 초기 로컬 구동용 FAISS에서 향후 클라우드 환경의 AWS Bedrock Knowledge Base로 매끄럽게 전환될 수 있도록 유연하게 구성합니다.
