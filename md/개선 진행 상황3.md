## 🧪 Step 1: 사용자 테스트 및 수동 검증 가이드

백엔드 DB 스키마 및 엔티티가 정상적으로 로드 및 동작하는지 아래의 두 가지 방법으로 직접 확인하실 수 있습니다.

### 방법 1. 자동화 통합 테스트 (JUnit) 구동
백엔드 소스코드에 삽입된 확장 테스트 클래스를 수행하여 데이터 정합성을 검증합니다.
1. 터미널(PowerShell 또는 CMD)에서 backend 폴더로 이동합니다:
   ```powershell
   cd c:\capd\yeonam_tester\backend
   ```
2. 다음 명령어를 실행하여 신규 추가된 `testTestCaseEntityExtension` 단위 테스트를 구동합니다:
   ```powershell
   mvn test -Dtest=Phase6ExtensionsTests#testTestCaseEntityExtension
   ```
3. **검증 기준**: JUnit 단언문(`assertEquals`)을 통해 가상의 테스트 케이스를 생성해 H2 DB에 저장했다가 다시 꺼냈을 때, 새로 이식한 4개 필드(`test_technique`, `Negative Testing Philosophy`, `TDD Hint text`, `Negative Scenario text`)가 유실 없이 일치하여 **BUILD SUCCESS**가 나타나는지 확인합니다.

### 방법 2. H2 Database 웹 콘솔을 통한 수동 물리 테이블 검증
1. 백엔드 Spring Boot 애플리케이션을 구동합니다:
   ```powershell
   cd c:\capd\yeonam_tester\backend
   mvn spring-boot:run
   ```
2. 웹 브라우저를 열어 H2 콘솔 주소(`http://localhost:8080/h2-console`)에 접속합니다.
3. JDBC URL 항목에 `jdbc:h2:mem:testerdb` (혹은 application.yml에 지정된 DB 명세)를 입력하고 **Connect** 버튼을 클릭합니다.
4. 좌측 테이블 리스트에서 `TEST_CASE`를 클릭하거나, 쿼리 입력 창에 아래 명령을 실행합니다:
   ```sql
   SELECT COLUMN_NAME, DATA_TYPE FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_NAME = 'TEST_CASE';
   ```
5. **검증 기준**: 쿼리 결과 목록에 신규 추가한 `CATEGORY`, `TECHNIQUE`, `TDD_HINT`, `NEGATIVE_SCENARIO` 컬럼이 데이터 타입과 함께 안전하게 등재되어 있는지 직접 눈으로 확인합니다.

## 🛠️ Step 4: 구현 완료 상세 내역 (Task 4.1)

사용자가 요구사항 문서를 업로드했을 때, 청킹/임베딩 단계 부재로 인해 파일 상태가 `UPLOADED`에 머무르는 고착 현상을 해결하기 위해 Spring Boot의 `@Async` 비동기 서비스와 FastAPI의 전처리 API 검증 연계를 완료하였습니다.

1. **Spring Boot 비동기 처리 가동**:
   - [TesterApplication.java](file:///c:/capd/yeonam_tester/backend/src/main/java/com/yeonam/tester/TesterApplication.java)에 `@EnableAsync`를 선언하여 비동기 실행 스레드 풀 구동 설정을 마쳤습니다.
2. **비동기 파일 전처리 서비스 구현**:
   - `FilePreprocessingService.java` 비동기 컴포넌트를 신설하여 `UPLOADED` 상태의 파일에 대해 `@Async`를 통해 `PROCESSING` 상태로 전이시킨 후, FastAPI 서버의 `/api/files/preprocess` API로 검증을 트리거하도록 구현하였습니다.
   - **오프라인/장애 복구 샌드박스 폴백**: AI 서버가 기동되지 않았거나 연동 장애가 발생할 경우, 백엔드 로컬에서 파일 실재성 및 파일명을 자체 검증하여 안전하게 파일 상태를 `DONE`으로 전이시키는 견고한 폴백 메커니즘을 추가하였습니다.
3. **업로드 파이프라인 비동기 연동**:
   - [FileService.java](file:///c:/capd/yeonam_tester/backend/src/main/java/com/yeonam/tester/service/FileService.java)의 `uploadFile` (멀티파트 업로드) 및 `uploadRawTextFile` (직접 업로드) 내 저장 완료 트랜잭션 직후 `filePreprocessingService.preprocessFile()`을 호출하여 비동기 스레드를 실행하고, 사용자 브라우저에는 `201 Created` 응답을 대기 없이 즉각 반환하도록 최적화하였습니다.
4. **FastAPI 전처리 엔드포인트 구현**:
   - `llm_server/main.py`에 `/api/files/preprocess` 라우터를 신설하여 S3 파일 경로로부터 문서 파싱 가능성 여부를 사전 검사하고, 성공 시 성공 상태를 리턴하도록 처리하였습니다.

## 🧪 Step 6: 사용자 테스트 및 수동 검증 가이드

아래의 방법으로 시각화 및 관리 보드 연동이 완벽하게 가동하는지 직접 확인할 수 있습니다.

### 방법 1. UI 결과 확인 및 TDD 아코디언 가동
1. 애플리케이션을 구동하여 임의의 프로젝트에서 AI 검증 분석을 가동합니다. (또는 샌드박스 오프라인 데모 모드로 분석 시작을 클릭합니다.)
2. 분석이 끝나 결과 시각화 페이지(`/analysis-demo`)로 넘어가면, 생성된 테스트 케이스 카드를 확인합니다.
3. **검증 기준**:
   - 카드 좌측 상단에 카테고리 뱃지(예: `test_technique`가 초록색 배지로 표시)가 제대로 렌더링되는지 확인합니다.
   - 카드 하단에 `[💡 TDD 개발 구현 가이드]` 및 `[⚠️ Happy Path 너머 예외 검증 시나리오]` 접이식 영역이 있는지 확인하고, 각각 클릭했을 때 부드럽게 펼쳐지며 내용이 잘 렌더링되는지 확인합니다.

### 방법 2. 대시보드 관리 보드 수정/저장 테스트
1. 대시보드 페이지의 '테스트 케이스 관리 보드' 탭으로 이동합니다.
2. 테이블 리스트에 새로 추가된 Category와 Technique 컬럼이 표시되고 있는지 확인합니다.
3. 특정 테스트 케이스의 '수정(연필 모양)' 버튼을 누릅니다.
4. 수정 모달창에서 카테고리를 변경하고, 설계 기법, TDD Hint 및 부정 시나리오 텍스트를 임의로 수정한 후 '저장' 버튼을 클릭합니다.
5. **검증 기준**: 모달이 닫힌 뒤 테이블 리스트와 다시 열었을 때의 모달창 폼에 내가 수정하여 저장한 Atlassian 상세 내용들이 안전하게 영구 저장되어 유지되는지 확인합니다.
