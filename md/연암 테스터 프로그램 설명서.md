# 연암 테스터 프로그램 설명서

## 1. 목적

* **학생 맞춤형 QA/TDD 학습 플랫폼:** 소프트웨어 전공 학생들이 자신의 프로젝트에서 직접 QA(품질 보증) 및 TDD(테스트 주도 개발) 사고방식을 학습하고 적용해 볼 수 있도록 돕는 보조 플랫폼입니다.
* **캡스톤 디자인 기반의 교육용 MVP:** 기업 실무 자동화 도구가 아닌, 학과 및 캡스톤 디자인 환경에서 학생이 프로젝트 품질을 점검하고 테스트 관점을 확보하도록 돕는 최소 기능 제품(MVP)입니다.
* **프로젝트 품질 점검 및 근거 제시:** AI가 실무자의 판단을 전면 대체하는 대신, 학생들이 놓치기 쉬운 비기능적 요구사항, 위험 요소 등을 짚어주며 QA적 관점의 테스트 초안을 도출하도록 보조합니다.

## 2. 기능

* **프로젝트 문서 전처리 및 추출:** 사용자가 요구사항 명세서, README, 설계 문서 등을 업로드하면, 시스템이 문서를 분석하여 핵심 기능과 요구사항을 구조화하여 추출합니다.
* **테스트 관점 및 TDD 초안 생성:** 추출된 요구사항을 바탕으로 사용자가 직접 선택한 관점에 맞춰 단위/통합/예외 테스트 수준의 TDD 테스트 시나리오 초안을 자동 생성합니다.
* **위험 요소 탐지 및 다중 근거 매핑 (1:N):** 인증 실패, 입력값 오류 등 발생 가능한 위험 유형을 제시하고, 도출된 테스트 케이스가 어떤 요구사항과 QA 지식을 근거로 작성되었는지 **최대 3개의 유효한 근거를 연결하는 1:N 방식으로 다중 매핑**하여 제공합니다.
* **TDD 검증 보고서 산출 (Markdown 우선):** 도출된 요약, 테스트 케이스, 근거 문장 등을 브라우저에서 즉시 검토할 수 있도록 렌더링하며, 최종적으로 **Markdown 또는 PDF 포맷의 검증 보고서** 형태로 자동 취합하여 다운로드할 수 있게 합니다.

## 3. 기술

* **프론트엔드 (React):** 사용자 흐름 제어, 파일 업로드, 결과의 시각적 렌더링(카드 UI) 및 Markdown 보고서 미리보기를 담당합니다.
* **백엔드 API 서버 (Spring Boot):** 사용자의 파일 업로드/다운로드 처리, 분석 작업 관리, S3 연동 및 데이터베이스(H2 DB 등)와의 메타데이터 통신을 담당합니다.
* **AI 및 분석 서버 (RAG Server):** 문서 텍스트 파싱, 청킹, 요구사항 추출, QA 지식 검색 및 LiteLLM 등 외부 LLM 호출을 통한 AI 분석 로직을 독립적으로 전담합니다.
* **데이터 저장소 (S3 & Vector DB):** 업로드 문서 및 생성된 보고서는 파일 저장소(S3 또는 로컬 경로)에 보관되며, 문서 조각 및 QA 지식베이스는 유사도 검색을 위해 **Vector DB (초기 MVP용 FAISS 및 향후 확장을 위한 AWS Bedrock Knowledge Base 지원 유연화 구조)**에 임베딩되어 저장됩니다.

## 4. 사용 시나리오

* **1단계 (정보 입력 및 문서 업로드):** 사용자는 웹 대시보드에 접속하여 **프로젝트명, 개요, 주요 기능, GitHub 주소 등 프로젝트 정보를 입력**하고, 요구사항 명세서나 설계 문서(PDF, MD, TXT, DOCX 등)를 업로드합니다.
* **2단계 (명시적 분석 시작 및 검색):** 자동 실행이 아닌, 사용자가 대시보드의 **'전체 분석 시작(또는 분석 시작)' 버튼을 명시적으로 클릭**하면 상태가 대기 및 진행 중으로 변경되며 백엔드 파이프라인이 구동됩니다. 이후 RAG를 활용해 선택된 QA 관점 기반의 관련 지식을 검색합니다.
* **3단계 (결과 렌더링 및 검토):** 분석이 완료되면 테스트 케이스 시나리오, 우선순위, 1:N으로 매핑된 근거 문장, 동적 위험 요소 태그가 직관적인 카드형 UI로 화면에 렌더링됩니다.
* **4단계 (보고서 미리보기 및 다운로드):** 사용자는 브라우저 상의 미리보기 패널을 통해 산출된 데이터를 검토하고, 최종적으로 구조화된 **Markdown 또는 PDF 검증 보고서**를 다운로드하여 캡스톤 프로젝트 산출물로 활용합니다.

## 5. 아키텍처 구성요소/구조

* **주요 구성요소 및 역할:**
  * **React Frontend:** 화면 구성, 사용자 입력, 프로젝트 정보 수집, QA 관점 선택, 분석 진행 상태 모니터링, 카드형 결과 조회.
  * **Spring Boot API Server:** 프로젝트 생성 API 제공, 파일 저장(S3 연계), RAG 분석 작업 생성 및 이력/상태 관리, Markdown 보고서 다운로드 처리.
  * **RAG Server:** 텍스트 추출(DocumentParser), 검색 질의 생성(QueryBuilder), QA 지식베이스 검색(Retriever), 프롬프트 동적 조립(PromptBuilder) 및 LLM 호출 처리.
  * **저장소 계층:** 원본 파일 및 문서 산출물 저장을 위한 **S3**, 텍스트 벡터 인덱싱과 유사도 탐색을 위한 **Vector DB**.
* **상호작용 시나리오:**
    1. 사용자가 React 프론트엔드를 통해 프로젝트 정보와 문서를 업로드하면, Spring Boot 백엔드는 S3에 파일을 안전하게 저장하고 H2 DB에 초기 상태를 기록합니다.
    2. 사용자가 '분석 시작'을 요청하면 Spring Boot가 RAG Server로 분석을 지시합니다.
    3. RAG Server는 문서를 논리적으로 분할(청킹)하여 Vector DB에 인덱싱하고, 시스템과 사용자의 QA 관점을 조합하여 관련 근거를 검색합니다.
    4. 검색된 근거(최대 3개)와 요구사항을 바탕으로 LLM을 호출하여 테스트 시나리오를 동적으로 조립 및 파싱합니다.
    5. 파싱된 결과는 Spring Boot를 통해 다시 React Frontend로 전달되어 카드 컴포넌트로 시각화되며, 사용자의 요청 시 Markdown 렌더링을 거쳐 보고서 파일로 제공됩니다.
