## 🛠️ Step 1: 구현 완료 상세 내역 (Task 1.1)

사용자가 프론트엔드에서 입력한 LLM API Key를 백엔드로 안전하게 전달하고, 이 Key가 최종적으로 AI 서버(FastAPI)의 litellm 추론 엔진까지 연계되어 동적 바인딩 및 사용이 가능하도록 데이터 아키텍처 파이프라인의 확장을 완수하였습니다.

1. **백엔드 DTO 및 서비스 연동**:
   - `AnalysisCreateRequest.java` DTO 파일에 `llmApiKey` 필드와 빌더 체인을 이식했습니다.
   - [AnalysisService.java](file:///c:/capd/yeonam_tester/backend/src/main/java/com/yeonam/tester/service/AnalysisService.java)의 비동기 AI 서버 트리거 로직을 수정하여, 수신한 `llmApiKey`를 JSON payload 바디에 태워 FastAPI(`/api/analysis/trigger` 및 `/analyze`)로 보내도록 구성했습니다.
2. **FastAPI Pydantic 규격 및 큐 흐름 이식**:
   - `llm_server/main.py`의 `TriggerRequest` 스키마 모델에 `llmApiKey: Optional[str] = None` 속성을 반영하고, asyncio background queue 작업 객체 사전에도 `"llmApiKey"` 데이터 필드를 저장하도록 보강했습니다.
3. **litellm 엔진 API Key 다이내믹 바인딩**:
   - `llm_server/llm_client.py`의 `call_llm` 메서드에서 `llm_api_key`를 넘겨받아 `litellm.acompletion` 호출 시 `api_key` 인자로 동적 매핑(예: `api_key=llm_api_key`)하게 가동하였습니다. 값이 없을 경우에는 디폴트로 환경변수(`os.getenv("OPENAI_API_KEY")`)를 참조하게 처리하여 유연성을 확보했습니다.

### 방법 1. 자동화 통합 테스트 (JUnit) 구동
1. 터미널에서 backend 폴더로 이동합니다:

2. 다음 명령어를 실행하여 커스텀 API Key 분석 요청 트리거 및 DTO 매핑 테스트 케이스를 가동합니다:

3. **검증 기준**: 테스트가 에러 없이 작동하고 `BUILD SUCCESS`가 나타나 백엔드의 DTO 및 서비스 파라미터 매핑 정합성이 깨지지 않았음을 판단합니다.

### 방법 2. AI 서버 cURL API 수동 호출 검증
1. `llm_server` 폴더로 이동하여 uvicorn 개발 서버를 구동합니다 (포트 8000).
2. 다른 터미널에서 아래 cURL 문장을 실행하여 수동으로 `llmApiKey`가 묻어 있는 분석 트리거 요청을 수행합니다:
   ```powershell
   curl -X POST "http://localhost:8000/api/analysis/trigger" -H "Content-Type: application/json" -d "{\"analysisId\":\"ANL-KEY-MOCK-1\",\"projectId\":\"PRJ-KEY-MOCK-1\",\"s3Paths\":[],\"llmApiKey\":\"sk-fake-user-key-value\"}"
   ```
3. **검증 기준**: HTTP 202 응답(`{"message": "Job accepted and queued for analysis."}`)을 정상 회신받고, uvicorn 서버 로그 상에서 작업 수신 및 litellm 트리거 준비 과정에 API Key가 정상 맵핑되는지 터미널의 log stream으로 육안 점검합니다.

## 🛠️ Step 2: 구현 완료 상세 내역 (Task 2.1)

대시보드 관리 보드에서 복수 선택된 테스트 케이스들을 취합하여 보고서를 생성할 때, 서로 다른 분석 작업(AnalysisJob)에 포함된 테스트 케이스들도 하나의 보고서로 정상 통합되도록 백엔드 로직을 보완했습니다.

1. **`ReportAssemblyService.java` 내 AnalysisId 일치 필터링 제거**:
   - `assembleReportData` 메서드에서 `testCaseIds` 리스트를 전달받아 테스트 케이스 객체들을 조회할 때, representative `analysisId`와 일치하지 않는 타 분석 작업의 테스트 케이스를 걷어내던 필터 루프 로직을 영구히 제거했습니다.
   - 이를 통해, 사용자가 UI 상에서 다르게 진행된 여러 분석 작업들의 테스트 케이스들을 체크 박스로 자유롭게 골라 하나의 보고서로 묶어 생성할 수 있게 되었습니다.
2. **`ReportService.java` 내 다중 매핑 영속화 검증**:
   - 대표 `analysisId`를 통해 보고서 엔티티를 생성하고, 매핑 테이블인 `ReportTestCase`에 전달받은 모든 테스트 케이스 ID와의 연결 레코드를 정상적으로 적재하도록 보완된 로직을 재확인했습니다.

2. 다음 명령어를 실행하여 다중 분석 작업 출신 테스트 케이스 통합 보고서 생성 테스트 케이스를 가동합니다:

### 방법 2. H2 데이터베이스 및 RDB 조회 수동 검증
1. 애플리케이션 백엔드를 기동한 후, `http://localhost:8080/h2-console`에 접속합니다. (JDBC URL: `jdbc:h2:file:./data/yeonam_db`, User Name: `sa`, Password: 없음)
2. Postman 등 API 테스터를 사용하여, 다른 분석 ID에 속한 테스트 케이스 ID 목록을 담아 보고서 생성 API를 호출합니다:
   - **Endpoint**: `POST http://localhost:8080/api/analysis/{analysisId}/reports`
   - **Body (JSON)**:
     ```json
     {
       "reportFormat": "MARKDOWN",
       "testCaseIds": ["TC-1", "TC-2"] // 서로 다른 분석 작업(AnalysisJob) 소속 케이스
     }
     ```
3. 생성된 이후 H2 SQL 콘솔에서 아래 쿼리를 수행하여 두 개의 레코드가 모두 하나의 `REPORT_ID`에 묶여 매핑되어 있는지 확인합니다:
   ```sql
   SELECT REPORT_ID, TEST_CASE_ID FROM REPORT_TEST_CASE;
   ```

## 🛠️ Step 3: 구현 완료 상세 내역 (Task 3.1)

좌측 사이드바 레이아웃 구조를 상단 헤더로 통합하고 불필요한 Support 항목을 영구 제거하였으며, 사용자 로컬 LLM API Key를 브라우저 단에서 마스킹 및 복사 차단 처리를 가미해 안전하게 저장 및 분석 트리거 시 자동으로 동적 주입할 수 있도록 구현을 완료하였습니다.

1. **상단 수평 네비게이션 탭 통합 및 사이드바 제거**:
   - `App.tsx`의 좌측 사이드바 JSX 마크업을 영구 제거하고 본문 너비가 사이드바 없이 가로 전체를 미려하게 활용하도록 `md:pl-64` 패딩 클래스를 걷어냈습니다.
   - 상단 헤더바에 `Dashboard` 및 `문서 업로드` 수평 네비게이션 `NavLink` 탭 링크를 이식하여 페이지 이동성을 간결하게 설계하였습니다.
2. **Support 삭제 및 API Key 설정 모달 창 구현**:
   - 기존 헤더 우측의 'Support' 메뉴 항목을 영구 삭제했습니다.
   - 우측 사용자 도구 영역에 'API Key 설정' 버튼을 추가하고, API Key 설정 상태에 따라 🔑 배지와 텍스트 색상이 초록색(`emerald-400` / `설정됨`) 또는 회색(`설정`)으로 반응하도록 구현하여 사용자 편의성을 높였습니다.
   - API Key 입력 팝업 모달창에서 입력 비밀번호 타입(`<input type="password" />`)을 적용하고 클립보드 복사를 차단(`onCopy={(e) => e.preventDefault()}`)하여 보안 위협을 예방하였고, `localStorage.setItem('yeonam_llm_api_key', key)`를 호출하여 브라우저에 캐싱 및 삭제 처리를 연계하였습니다.
3. **업로드 분석 시작 시 API Key 자동 바인딩 연동**:
   - `DocumentUploadPage.tsx`의 '분석 시작' 동작인 `handleStartAnalysis` 내부에서 `localStorage.getItem('yeonam_llm_api_key') || ''` 값을 읽어 DTO 파라미터 `llmApiKey`에 자동으로 동적 바인딩하여 백엔드로 전달하도록 구현을 마쳤습니다.

3. **백엔드 연쇄 파기 트랜잭션 연동**:
   - `AnalysisService.java`에 `@Transactional` 및 `deleteAnalysisJob` API를 통해 RDB 연관 엔티티의 무결한 연쇄 삭제 및 S3 물리 리포트 Object 소멸 연동 처리를 완료하였습니다.
