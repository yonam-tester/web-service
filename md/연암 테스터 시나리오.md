### 시나리오 1. RAG 기반 파이프라인 통합 분석 및 검증 보고서 산출 정상 흐름
본 시나리오는 복합 문서를 업로드하고 전체 RAG 파이프라인을 거쳐 테스트 케이스를 생성한 뒤, 최종 보고서를 다운로드하는 가장 핵심적인 표준 흐름입니다.

**1.1. 프로젝트 셋업 및 초기 환경 구성**
1.1.1 * 사용자는 프로젝트 기본 정보 입력 폼에 프로젝트명, 개요, 주요 기능을 기재합니다. `(1.1.1)`
1.1.2 * 입력한 GitHub 주소는 브라우저와 서버 단에서 유효성 검사 및 리포지토리 접근 확인을 거칩니다. `(1.1.1.1)`
1.1.3 * 유효할 경우 GitHub 메타데이터 및 README 문서를 자동 수집하고 `(1.1.1.2)`, 향후 확장을 위한 코드 저장소 연동 설정 정보가 H2 DB에 저장됩니다. `(1.1.1.3)`
1.1.4 * 설정 완료 후 통합 프로젝트 대시보드 화면이 제공되며 `(1.1.5.1)`, 전체 프로젝트와 분석 작업 단위 관리가 시작됩니다. `(1.1.5)`

**1.2. 문서 업로드 및 무결성 검증**  
1.2.1 * 대시보드에서 분석 대상을 선택해 업로드하면 파일 포맷 및 용량 유효성 검증이 진행됩니다. `(1.1.3)` 
1.2.2 * 프론트엔드에서 즉시 20MB 제한 및 확장자를 사전 검사해 부적합 시 차단하고 `(1.1.3.1)`, 서버 진입 시 빈 파일 여부 등 백엔드 무결성 검증 및 예외 반환을 거칩니다. `(1.1.3.2)` 
1.2.3 * 검증을 통과한 파일은 사용자가 생성한 프로젝트를 기반으로 S3 연동 처리를 통해 물리적으로 안전하게 저장되며, 업로드된 파일은 해당 프로젝트에 포함됩니다. `(1.1.2)` 
1.2.4 * 분석을 위해 문서 메타데이터가 파싱되며 초기 상태(UPLOADED 등)가 데이터베이스에 저장됩니다. `(1.1.4)`

**1.3. 문서 파싱, 정제 및 논리적 분할(청킹)**
1.3.1 * PDF/DOCX 등의 복합 문서 파싱 로직이 가동됩니다. `(1.3.2)`
    1.3.1.1 * 이미지나 다단이 섞인 문서 레이아웃을 분석해 의미 단락을 추출합니다. `(1.3.2.1)`
    1.3.1.2 * 서식 및 계층 등 문서 구조 및 스타일 속성을 유지하며 `(1.3.2.2)`, 표(Table) 데이터를 구조화 포맷(Markdown)으로 변환합니다. `(1.3.2.3)`
1.3.2 * 추출된 텍스트의 불순물을 정제하고 필터링합니다. `(1.4.1)`
    1.3.2.1 * 머리글/바닥글 등 레이아웃 노이즈를 제거하고 `(1.4.1.1)`, 공백과 특수문자 등 텍스트 포맷팅을 정규화합니다. `(1.4.1.2)`
    1.3.2.2 * 분석 의미가 없는 짧은 단문을 필터링 및 병합합니다. `(1.4.1.3)`
1.3.3 * 정제된 텍스트를 문서 구조 기반으로 논리적으로 분할합니다. `(1.5.1)`
    1.3.3.1 * 계층적 제목을 기준으로 의미 단락을 1차 분할하고 `(1.5.1.1)`, 구조화 데이터(표, 목록)는 중간에 끊기지 않게 무결성을 보존해 분할합니다. `(1.5.1.2)`
    1.3.3.2 * 임베딩 최적화 크기로 분할(청킹)을 진행하며 `(1.5.2)`, 문맥 단절 방지를 위해 오버랩(Overlap) 분할을 적용하고 `(1.5.2.1)`, 청크별 컨텍스트 메타데이터(상위 제목 등)를 병합합니다. `(1.5.2.2)`

**1.4. 임베딩, 요구사항 추출 및 분석 트리거**
1.4.1 * 분할된 청크의 임베딩 생성 및 벡터 인덱싱이 수행됩니다. `(1.6.1)`
    1.4.1.1 * 텍스트 청크를 다차원 벡터(Embedding) 수치 데이터로 변환합니다. `(1.6.1.1)`
    1.4.1.2 * 추적을 위해 원본 식별자 등 메타데이터와 결합하여 Vector DB에 인덱스로 저장합니다. `(1.6.1.2)`
1.4.2 * 원본 문서 문맥에서 시스템 핵심 기능 및 사용자 흐름을 식별합니다. `(1.7.1)`
1.4.3 * 이 중 검증 가능한 개별 요구사항 단위만을 노이즈 없이 분리 및 추출합니다. `(1.7.2)`
1.4.4 * 추출 완료된 요구사항에 고유 식별자(ID)를 부여해 DB 메타데이터로 기록합니다. `(1.7.3)`
1.4.5 * 문맥을 기반으로 시스템이 맞춤형 QA 관점을 대시보드에 동적 추천합니다. `(2.1.1.2)`
1.4.6 * 사용자는 보안, 성능 등 사전 정의된 기본 QA 관점을 명시적으로 선택합니다. `(2.1.1, 2.1.1.1)`
1.4.7 * 선택에 따라 분석 프롬프트의 동적 관점 매개변수가 조정되고 `(2.1.2)`, 시스템 내부의 프롬프트 엔지니어링 관점이 자동 삽입됩니다. `(2.1.3)`
1.4.8 * 향후 동일한 환경을 불러오기 위해 사용자 분석 조건이 재현 가능하게 DB에 저장됩니다. `(2.1.4)`
1.4.9 * 사용자가 컨텍스트 인식형 원클릭 파이프라인 제어기로 분석을 지시합니다. `(4.2.1)`
1.4.10 * 백엔드는 비동기 작업 대기열(Queue)에 이를 위임하며 `(2.2.3.1)`, 화면에는 파이프라인 단계별 진행 상태를 알려주는 스피너 등 모니터링 UI가 렌더링됩니다. `(4.1.1)`

**1.5. 교차 검색(Cross-Search) 및 프롬프트 조립**
1.5.1 * RAG 파이프라인이 가동되어 벡터 유사도 기반 관련 문서 청크를 검색합니다. `(3.1.1)`
    1.5.1.1 * 검색 질의(Query)가 문맥 임베딩 벡터로 변환됩니다. `(3.1.1.1)`
    1.5.1.2 * Vector DB를 탐색해 Top-K 유사도 탐색을 진행하고 `(3.1.1.2)`, 기준치(Threshold) 미달 데이터는 필터링하여 검색 품질을 제어합니다. `(3.1.1.3)`
1.5.2 * 시스템 내부에 사전 구축된 'QA 지식베이스' 인덱스 공간을 연동합니다. `(3.1.2, 3.1.2.1)`
1.5.3 * 요구사항 문맥을 기반으로 지식베이스 내 결함 사례 등을 교차 검색하고 `(3.1.2.2)`, 사용자 문서 근거와 지식베이스 검색 결과를 가중치에 따라 병합합니다. `(3.1.2.3)`
1.5.4 * 도출된 검색 근거를 바탕으로 동적 프롬프트를 조립합니다. `(2.2.1)`
1.5.5 * 시스템 지시문과 구조화 데이터를 식별 태그로 주입하고 `(2.2.1.1)`, LLM 컨텍스트 윈도우(Token) 초과 방지 최적화 단계를 거칩니다. `(2.2.1.2)`

**1.6. RAG 통신, 파싱 및 우선순위/위험 요소 도출**
1.6.1 * AI 분석 계층의 플러그 앤 플레이 모드에 따라 `(2.2.2)`, RAG 서버(고도화 구동 모드)가 연결되어 외부 LLM과 통신합니다. `(2.2.2.2)`
1.6.2 * 분석이 끝나면 웹훅(Webhook) 규격을 통해 구조화 결과 통합 콜백을 전송/수신합니다. `(2.2.3, 2.2.3.2)`
1.6.3 * LLM 응답 데이터의 구조화 파싱이 진행됩니다. `(2.3.1)`
    1.6.3.1 * 응답 중 지정 포맷(JSON/Markdown 표)만 식별하여 추출하고 `(2.3.1.1)`, 해당 데이터를 테스트 케이스 등 QA 세부 항목 객체로 매핑 파싱합니다. `(2.3.1.2)`
    1.6.3.2 * 파싱 완료된 생성 결과 단위를 데이터베이스에 저장합니다. `(2.3.2)`
1.6.4 * 시스템이 테스트 케이스 다면적 우선순위를 자동 산정합니다. `(2.4.1)`
    1.6.4.1 * 요구사항 중요도 및 사용자 영향도 분석을 진행하고 `(2.4.1.1)`, 오류 발생 가능성 가중치를 산정하여 `(2.4.1.2)`, 종합 우선순위 등급(높음/중간/낮음)을 할당합니다. `(2.4.1.3)`
1.6.5 * 요구사항 문맥을 분석하여 동적으로 핵심 결함(위험) 유형을 식별합니다. `(2.5.1, 2.5.1.1)`
1.6.6 * 도출된 위험 요소는 식별 해시태그(Tag)로 변환되어 컴포넌트에 부착할 수 있도록 데이터베이스에 매핑됩니다. `(2.5.1.2)`
1.6.7 * 테스트 케이스와 다중 근거(1:N) 객체를 매핑합니다. `(3.2.2)`
    1.6.7.1 * 검색 근거 메타데이터를 구성해 유사도를 평가하고 `(3.2.1)`, 최적 근거 선별을 거쳐 참조 개수를 최대 3개로 제한합니다. `(3.2.2.1)`
    1.6.7.2 * 테스트 케이스에 요구사항-근거 1:N 추적성(Traceability) 객체를 결합하고 `(3.2.2.2)`, 매핑 무결성을 위해 데이터베이스(DB) 트랜잭션 기록을 수행합니다. `(3.2.2.3)`

**1.7. 시각적 UI 렌더링 및 검증 보고서 반출**
1.7.1 * 대시보드 화면에 분석 결과를 단계별 카드형으로 시각화합니다. `(4.1.2)`
    1.7.1.1 * 최상단에 요구사항 명세서 전체 요약 카드를 렌더링하고 `(4.1.2.1)`, 각 테스트 케이스의 세부 항목(절차 등)을 구조화된 상세 컴포넌트로 그립니다. `(4.1.2.2)`
    1.7.1.2 * 각 카드마다 도출된 우선순위 배지 및 위험 요소 태그가 시각적으로 강조 렌더링됩니다. `(4.1.2.3)`
    1.7.1.3 * 장문 근거 데이터의 텍스트 길이 감지 시 말줄임표(Truncation)가 자동 적용되고 `(3.4.2, 3.4.2.1)`, 아코디언(Accordion) 방식의 원문 확장/축소 토글 UI가 상호작용으로 제공됩니다. `(3.4.2.2)`
1.7.2 * 보고서 다운로드 요청 시 도메인 데이터를 종합 및 구조화합니다. `(4.3.1)`
    1.7.2.1 * 프로젝트 개요 및 명세서 요약 정보를 취합하고 `(4.3.1.1)`, 상세 시나리오 객체와 1:N 다중 근거 데이터를 병합합니다. `(4.3.1.2)`
1.7.3 * 취합 중 '보완 필요 사항' 및 '한계/주의사항' 자동 도출 기능이 병합 실행됩니다. `(4.3.5)`
    1.7.3.1 * 명세서 대비 보완 필요 사항을 동적으로 추론하고 `(4.3.5.1)`, 생성 결과 한계 고지(Disclaimer) 템플릿을 자동 삽입하여 `(4.3.5.2)`, 보고서 하단에 독립 목차로 할당하고 통합 렌더링을 준비합니다. `(4.3.5.3)`
1.7.4 * 최종 조립된 데이터 모델(DTO)의 무결성 누락 검증을 마친 후 `(4.3.1.3)`, Markdown 및 PDF 포맷 문서 렌더링 변환이 진행됩니다. `(4.3.2)`
    1.7.4.1 * 템플릿에 맞추어 Markdown 검증 보고서를 생성 조립하고 `(4.3.2.1)`, 시각화 양식의 PDF 검증 보고서로 렌더링 변환합니다. `(4.3.2.2)`
1.7.5 * 생성 완료된 보고서 파일은 S3에 물리적으로 저장되며 `(4.3.3, 4.3.3.1)`, 이 보고서 생성 이력 메타데이터는 안전하게 DB에 기록됩니다. `(4.3.4, 4.3.4.1)`
1.7.6 * 브라우저 기반 보고서 렌더링(미리보기 UI)으로 화면 이탈 없이 최종 확인이 가능합니다. `(4.4.1)`
1.7.7 * 사용자가 최종 다운로드를 클릭하면 물리적 파일 존재 여부 무결성 검증 후 `(4.3.3.2)`, 정상적으로 클라이언트 반출이 완료됩니다. `(4.5.1)`

### 시나리오 2. 특수 지시, 예외 방어 및 무중단 자동 복구 흐름
사용자가 단순 파일을 올리거나 시스템 내부적으로 네트워크 및 LLM 환각 문제가 발생했을 때, 애플리케이션 중단을 방어하고 내부적으로 안전하게 처리하는 흐름입니다.

**2.1. 단순 텍스트 파싱 및 사용자 지정 지시**
2.1.1 * 복잡한 PDF가 아닌 구조가 단순한 문서를 올릴 경우 텍스트 문서(TXT, MD) 파싱 기능으로 추출이 진행됩니다. `(1.3.1)`
2.1.2 * 분석 요청 시 사용자가 프롬프트에 직접 테스트 통제 조건을 지시할 수 있는 '사용자 지정 프롬프트 입력 기능'을 활용할 수 있습니다. `(2.1.5)`
2.1.3 * 가벼운 테스트의 경우 무거운 RAG 대신 LLM 서버 구동 모드(MVP 모드)로 교체 스위칭하여 구동됩니다. `(2.2.2.1)`

**2.2. 누락 데이터 및 통신 에러 내부 방어**
2.2.1 * 비동기 작업 중 RAG 서버 등과의 통신 에러 발생 시, 시스템이 다운되는 대신 사용자 친화적 예외 처리로 파이프라인을 안전하게 종료하고 실패 상태를 응답합니다. `(2.2.3.3)`
2.2.2 * LLM 응답 구조 파괴로 '기대 결과' 등 필수 데이터 누락 감지 시 에러를 뱉는 대신, 예외(Fallback) 방어 기능이 가동되어 내부적으로 '추가 검토 필요' 기본값을 채워넣어 부분 성공 처리합니다. `(2.3.1.3)`

**2.3. 환각 인덱스 식별 및 무중단 데이터 복구**
2.3.1  매핑 단계에서 테스트 케이스 근거 인덱스 검증 기능이 작동합니다. `(3.3.1)`
    2.3.1.1 * 반환된 인덱스가 존재하는지 유효 범위 이탈(Out-of-Bounds) 여부를 검사합니다. `(3.3.1.1)`
    2.3.1.2 * 무효 인덱스로 판별(환각)될 경우, 잘못된 매핑을 강제하지 않고 신뢰성 확보(Fallback) 차원에서 해당 근거를 제외합니다. `(3.3.1.2)`
    2.3.1.3 * 이 방어 이력은 환각 빈도 추적을 위해 서버 시스템 로그로 상세 로깅됩니다. `(3.3.1.3)`
2.3.2 * 그 결과 UI 상에서는 해당 케이스 카드에 근거 부족 시각적 경고가 렌더링됩니다. `(3.4.1)`
2.3.3 * 이전에 생성했던 케이스 상세 내역은 생성 이력 단건 조회 기능으로 언제든 다시 볼 수 있습니다. `(2.3.3)`
2.3.4 * 사용자가 과거 보고서 다운로드 시 스토리지에서 파일이 유실되었더라도, 시스템이 DB 메타데이터를 기반으로 즉시 파일 백그라운드 자동 재생성(무중단 복구)을 가동하여 사용자 화면 에러 없이 무사히 다운로드를 완료합니다. `(4.3.3.3)`

### 시나리오 3. 데이터 생애주기 관리 및 연쇄 완전 파기(Cascade Delete) 흐름
업로드된 데이터와 산출물을 대시보드 상에서 조회하고, 정보 보안을 위해 원본 삭제 시 파생된 인덱스 데이터까지 연쇄적으로 파기하는 흐름입니다.

**3.1. 통합 이력 다중 조회 및 파일 제어**
3.1.1 * 프로젝트 대시보드에서 문서 목록 및 상태 조회가 가능하며 `(1.2.1)`, 보관 중인 문서 원본 다운로드를 지원합니다. `(1.2.2)`
3.1.2 * 원본 문서 번호(documentId)나 분석 단위(generationId)를 검색 조건으로 넣어 원본 문서 및 생성 이력 기반 다중 필터링 조회를 수행합니다. `(4.3.4.2)`

**3.2. 문서 영구 삭제 및 벡터 인덱스 파생 데이터 동기화 파기**
3.2.1 * 사용자가 특정 프로젝트 삭제 요청 시 해당 프로젝트의 모든 연관 데이터 연쇄 파기가 제어되어 전역 삭제가 발생합니다. `(1.1.5.2)`
3.2.2 * 특정 문서를 지정해 삭제 시도 시 파생 데이터 파기를 알리는 삭제 경고 모달 UI가 프론트엔드에 표출됩니다. `(1.2.3, 1.2.3.1)`
3.2.3 * 동의 후 승인하면, 백엔드에서 S3 원본 파일 및 메타데이터 삭제 처리가 수행됩니다. `(1.2.3.2)`
3.2.4 * 이와 동기화되어 Vector DB에 인덱싱 되어있던 해당 문서의 파생 데이터(텍스트 청크 및 임베딩 벡터)까지 추적하여 연쇄 완전 파기를 진행합니다. `(1.2.3.3)`

**3.3. 보고서 물리 파일 동기화 완전 파기**
3.3.1 * 과거 검증 보고서를 삭제할 경우, 단순 DB 메타데이터만 지우지 않고 실제 저장소(S3)에 보관된 마크다운/PDF 파일 원본까지 동기화하여 연쇄 파기하는 보고서 단건 삭제 기능을 처리해 저장소 용량을 안전하게 확보합니다. `(4.3.4.3)`
