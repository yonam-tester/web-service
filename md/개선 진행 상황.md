### 📘 Step 2 사용자 테스트 가이드

#### 1. API 다중 필터링 검증 (PowerShell / Postman)
1. Spring Boot 백엔드 서버(8080 포트)를 실행합니다.
2. 특정 프로젝트에 존재하는 보고서 중 `fileId`를 기준으로 필터링하기 위해 아래 PowerShell 명령어를 통해 GET 요청을 보냅니다:
   ```powershell
   # 1단계: 특정 프로젝트의 전체 보고서 조회
   Invoke-RestMethod -Uri "http://localhost:8080/api/projects/{projectId}/reports" -Method Get

   # 2단계: fileId 쿼리 파라미터를 추가하여 특정 문서에 매핑된 보고서만 필터링 조회

   # 3단계: analysisId 쿼리 파라미터를 추가하여 특정 분석에 매핑된 보고서만 필터링 조회
   Invoke-RestMethod -Uri "http://localhost:8080/api/projects/{projectId}/reports?analysisId={analysisId}" -Method Get
   ```
3. 각 쿼리 파라미터를 지정했을 때, 지정한 `fileId` 또는 `analysisId`를 갖는 보고서들만 응답 결과 리스트(`reports`)에 정확히 필터링되어 반환되는지 확인합니다.

## 🛠️ Step 1: 구현 완료 상세 내역 (Task 1.1)

사용자가 프론트엔드에서 입력한 LLM API Key를 백엔드로 안전하게 전달하고, 이 Key가 최종적으로 AI 서버(FastAPI)의 litellm 추론 엔진까지 연계되어 동적 바인딩 및 사용이 가능하도록 데이터 아키텍처 파이프라인의 확장을 완수하였습니다.

1. **백엔드 DTO 및 서비스 연동**:

   - `llm_server/main.py`의 `TriggerRequest` 스키마 모델에 `llmApiKey: Optional[str] = None` 속성을 반영하고, asyncio background queue 작업 객체 사전에도 `"llmApiKey"` 데이터 필드를 저장하도록 보강했습니다.
3. **litellm 엔진 API Key 다이내믹 바인딩**:

#### ➕ 추가 개선: S3 메타데이터 기반 완전한 프로젝트 정보 복구 기능 반영

사용자가 DB 초기화 후 S3(MinIO)로부터 데이터를 복구할 때 임시 명칭(예: `복구된 프로젝트_ID`) 대신, S3 객체에 저장된 **실제 프로젝트의 상세 정보(이름, 설명, GitHub URL, 브랜치 등)**를 온전히 복원할 수 있도록 기능을 고도화하였습니다.

###### 🛠️ 추가 구현 내역

1. **S3 파일 업로드 시 프로젝트 메타데이터 영속화:**
   - **대상 파일:** [FileService.java](file:///c:/capd/yeonam_tester/backend/src/main/java/com/yeonam/tester/service/FileService.java)
   - **변경 사항:**
     - 파일 업로드 및 GitHub 마크다운 업로드(`uploadFile`, `uploadRawTextFile`) 시, `PutObjectRequest`에 프로젝트의 모든 정보(`project-id`, `project-name`, `project-description`, `project-github-url`, `project-github-branch`, `project-integration-status`, `project-created-at`)를 User Metadata로 설정하여 S3에 저장합니다.
     - 한글 이름 및 설명이 깨지지 않도록 `URLEncoder.encode`를 사용해 UTF-8 아스키 형식으로 안전하게 인코딩하여 저장합니다.

2. **동기화 복구 시 S3 메타데이터 복원 처리:**
   - **대상 파일:** [S3SyncService.java](file:///c:/capd/yeonam_tester/backend/src/main/java/com/yeonam/tester/service/S3SyncService.java)
   - **변경 사항:**
     - S3 Object를 순회할 때, `s3Client.headObject()`를 호출하여 객체의 User Metadata를 읽어옵니다.
     - 메타데이터 정보가 존재할 경우, `URLDecoder.decode`를 거쳐 한글 데이터와 기존 프로젝트의 제반 설정들을 파싱하여 복구 엔티티에 통째로 바인딩합니다.
     - 만약 메타데이터가 존재하지 않는 이전 객체일 경우, 기존 방식대로 하위 호환성을 유지하기 위한 Fallback 기본값(`복구된 프로젝트_ID`)을 할당합니다.
