## 📘 Step 2 사용자 테스트 가이드

### 1. API 다중 필터링 검증 (PowerShell / Postman)
1. Spring Boot 백엔드 서버(8080 포트)를 실행합니다.
2. 특정 프로젝트에 존재하는 보고서 중 `fileId`를 기준으로 필터링하기 위해 아래 PowerShell 명령어를 통해 GET 요청을 보냅니다:
   ```powershell
   # 1단계: 특정 프로젝트의 전체 보고서 조회
   Invoke-RestMethod -Uri "http://localhost:8080/api/projects/{projectId}/reports" -Method Get

   # 2단계: fileId 쿼리 파라미터를 추가하여 특정 문서에 매핑된 보고서만 필터링 조회

   # 3단계: analysisId 쿼리 파라미터를 추가하여 특정 분석에 매핑된 보고서만 필터링 조회
   Invoke-RestMethod -Uri "http://localhost:8080/api/projects/{projectId}/reports?analysisId={analysisId}" -Method Get
   ```
3. 각 쿼리 파라미터를 지정했을 때, 지정한 `fileId` 또는 `analysisId`를 갖는 보고서들만 응답 결과 리스트(`reports`)에 정확히 필터링되어 반환되는지 확인합니다.

  - '문서 업로드 및 전처리 현황' 테이블의 각 명세서 행 파일명 옆에 깔때기 모양의 필터 아이콘(`filter_alt` Material symbol) 버튼을 배치.
  - 아이콘 클릭 시 `activeFilterFileId`를 지정하고, 해당 변경 감지(`useEffect`)가 트리거되면 `reportApi.getByProject(selectedProjectId, activeFilterFileId)`를 호출하여 하단의 보고서 이력 상태(`reports`)를 해당 문서와 연계된 결과로만 서버 필터 데이터로 갱신 렌더링.
  - 사용자가 필터링 상태를 시각적으로 직관적으로 알 수 있도록 보고서 이력 헤더 부분에 '필터 적용 중: [파일명] (X)' 배지 표시 영역을 추가하고 배지의 X 버튼 클릭 시 필터 해제 상태로 자동 복구되도록 구현 완료.
  - 필터가 설정된 상황에서 결과가 없을 때 "이 문서에 대해 생성된 보고서 이력이 없습니다."라는 문구를 노출하여 정보 가독성 증대.

## 📘 Step 4 사용자 테스트 가이드

실제 사용자가 웹 대시보드 화면을 통해 필터링 동작을 직접 수동 검증할 수 있는 가이드입니다.

### 1. 프론트엔드-백엔드 연쇄 필터링 수동 테스트
1. React 개발 서버(`http://localhost:5173`)와 Spring Boot 백엔드 서버(8080 포트)를 실행합니다.
2. 웹 브라우저에서 `http://localhost:5173` 대시보드 페이지에 진입합니다.
3. 임의의 기획서 문서 파일들이 업로드되어 있고, 이들을 기반으로 생성된 보고서들이 최소 2개 이상 존재하는 프로젝트를 선택합니다.
4. '문서 업로드 및 전처리 현황' 테이블에 노출되는 문서들 중 특정 문서 행의 **깔때기 아이콘**을 클릭합니다.
5. 클릭 시 하단 '분석 및 보고서 이력' 목록 헤더 옆에 **[필터 적용 중: 파일명]** 파란색 배지가 생겨나며 목록이 해당 문서로 분석해 만들어진 보고서로만 실시간 갱신되어 필터링되는지 확인합니다.
6. 배지 안의 **X (닫기) 버튼**을 누르거나 깔때기 아이콘을 한 번 더 누르면 필터가 즉각 해제되고 다시 전체 보고서 목록으로 복원되는지 확인합니다.
