###### 💡 연암 테스터 기능 구체 목록 (계층 구조화 반영본)
###### 1. 프로젝트 초기 설정 및 문서 전처리
*   **1.1. 프로젝트 정보 입력 및 문서 업로드 검증:**  프로젝트 기초 맥락을 수집하고, 문서 파일을 안전하게 서버로 전송 및 검사하는 단계입니다.
    *   **1.1.1. 프로젝트 기본 정보 입력 폼 제공:**  문서 업로드 전, 프로젝트명, 개요, 주요 기능, GitHub 주소 등 프로젝트의 기본 메타 정보를 입력하고 저장할 수 있는 기능.
        *   **1.1.1.1. GitHub 주소 유효성 검사 및 리포지토리 접근 확인:**  브라우저에서 올바른 URL 형식인지 검사하고, 서버 단에서 API를 통해 실제 접근 가능한 공개(Public) 리포지토리인지 확인하여 오류 시 알림을 제공하는 기능.
        *   **1.1.1.2. GitHub 메타데이터 및 README 문서 자동 수집:**  유효한 리포지토리의 README.md 파일과 프로젝트 기본 설명 등 메타데이터를 자동 수집하여 전처리 대상 문서로 파이프라인에 추가하는 기능.
        *   **1.1.1.3. 코드 저장소 연동 설정 저장:**  향후 구현 기반 테스트 자동 생성 확장에 대비해 브랜치 정보나 연동 상태 값을 H2 데이터베이스에 매핑하여 저장하는 기능.
    *   **1.1.2. 파일 업로드 및 S3 연동 처리:**  사용자가 파일을 업로드할 때 생성한 프로젝트를 기반으로 선택한 파일(pdf, md, txt, docx)을 서버로 전송받아 S3 또는 지정된 클라우드 저장소에 안전하게 업로드하며, 업로드된 파일은 해당 프로젝트에 포함되도록 하는 기능.
    *   **1.1.3. 파일 포맷 및 용량 유효성 검증:**  서버 진입 전후로 20MB 용량 제한, 허용 확장자, 빈 파일 여부를 검사하고 실패 사유를 반환하는 기능.
        *   **1.1.3.1. 프론트엔드 사전 유효성 검사 및 차단:**  파일 선택 즉시 브라우저 단에서 용량 및 확장자를 확인하여 부적합 시 경고 토스트 메시지를 띄우고 차단하는 기능.
        *   **1.1.3.2. 백엔드 무결성 검증 및 예외 반환:**  프론트엔드를 우회하여 들어온 비정상 요청이나 빈 파일 여부를 서버에서 재검증하고 명확한 실패 사유를 반환하는 기능.
    *   **1.1.4. 문서 메타데이터 파싱 및 상태 DB 저장:**  검증을 통과한 원본 파일의 S3 저장 경로와 파일명, 초기 처리 상태(NOT_STARTED 등) 메타데이터를 H2 데이터베이스에 기록하는 기능.
    *   **1.1.5. 프로젝트 및 분석 단위(AnalysisJob) 통합 관리 및 조회:**  단순 문서 정보 기록을 넘어, 사용자가 등록한 전체 프로젝트 목록을 조회하고 각 프로젝트별로 실행된 여러 분석 요청(AnalysisJob)의 진행 상태와 개별 결과물을 추적 및 관리하는 API 계층 기능.
        *   **1.1.5.1. 통합 프로젝트 대시보드 화면 제공:** 생성한 프로젝트를 한 화면에서 목록 형태로 확인하고, 각 프로젝트의 기본 정보 및 분석 진행 상태를 통합하여 관리할 수 있는 UI 제공 기능.
        *   **1.1.5.2. 프로젝트 삭제 및 연관 데이터 연쇄 파기 제어:** 통합 대시보드 화면에서 특정 프로젝트를 선택하여 삭제할 수 있으며, 삭제 시 해당 프로젝트에 연결된 업로드 문서 원본, 분석 요청, 테스트 케이스, 위험 요소, 보고서 등 모든 관련 데이터를 연쇄적으로 완전 파기하는 기능.
*   **1.2. 문서 관리 및 삭제 제어:**  업로드된 문서 목록을 관리하고, 파생 데이터를 포함한 안전한 문서 단위 삭제를 처리하는 단계입니다.
    *   **1.2.1. 문서 목록 및 상태 조회:**  업로드된 전체 문서의 목록과 전처리 진행 상태 등을 대시보드에서 조회하는 기능.
    *   **1.2.2. 문서 원본 다운로드:**  S3 등 서버 저장소에 보관된 특정 문서 원본 파일을 로컬로 다운로드하는 기능.
    *   **1.2.3. 문서 삭제 및 연쇄 파기:**  문서 삭제 시 원본 파일과 파생된 텍스트 청크, 벡터 임베딩 데이터까지 연쇄적으로 지우는 기능.
        *   **1.2.3.1. 문서 삭제 경고 모달 UI 제공:**  사용자가 삭제를 시도할 때 파생된 데이터까지 영구 삭제됨을 알리고 최종 확인을 받는 프론트엔드 기능.
        *   **1.2.3.2. S3 원본 파일 및 메타데이터 삭제 처리:**  H2 DB에서 문서 상태 및 메타데이터를 지우고, S3 저장소에 보관된 원본 파일을 삭제하는 백엔드 기능.
        *   **1.2.3.3. Vector DB 파생 데이터 연쇄 파기:**  문서 파싱으로 생성된 텍스트 청크와 임베딩 벡터 인덱스를 찾아 완전히 폐기하는 데이터베이스 연동 기능.
*   **1.3. 문서 파싱 및 텍스트 추출:**  포맷별로 다른 원본 문서 구조를 보존하며 분석 대상 텍스트를 추출하는 단계입니다.
    *   **1.3.1. 텍스트 문서(TXT, MD) 파싱 기능:**  구조가 단순한 텍스트나 마크다운의 구조를 보존하며 텍스트를 추출하는 기능.
    *   **1.3.2. 복합 문서(PDF, DOCX) 파싱 기능:**  이미지나 복잡한 레이아웃이 섞여 있는 문서에서 텍스트를 읽어오는 기능.
        *   **1.3.2.1. 문서 레이아웃 분석 및 단락 추출 기능:**  다단, 이미지 등이 섞인 복잡한 레이아웃 속에서 텍스트의 논리적인 흐름을 파악하여 순차적인 단락(Paragraph) 단위로 텍스트를 읽어오는 기능.
        *   **1.3.2.2. 문서 구조 및 스타일 속성 유지 기능:**  제목, 절, 목록 등 작성자가 지정한 서식 및 계층 정보를 인식하여, 텍스트 추출 시 해당 문서 구조를 가능한 범위에서 유지하는 기능.
        *   **1.3.2.3. 표(Table) 데이터의 구조화 변환 기능:**  문서 내에 포함된 요구사항이나 기능 명세 표를 인지하고, 행과 열의 데이터가 섞이지 않도록 Markdown 표 등의 구조화된 텍스트로 변환하여 추출하는 기능.
*   **1.4. 추출 텍스트 정제:**  추출된 텍스트의 불순물을 제거하여 분석 품질을 높이는 단계입니다.
    *   **1.4.1. 추출 텍스트 정제 및 필터링:**  추출된 텍스트에서 분석에 방해가 되는 노이즈를 제거하는 기능.
        *   **1.4.1.1. 문서 레이아웃 노이즈(머리글/바닥글/페이지 번호) 식별 및 제거 기능:**  페이지가 넘어갈 때마다 반복되는 문서의 머리글, 바닥글, 페이지 번호 등을 정규식이나 위치 정보로 식별하여 잘라냄으로써 문맥이 중간에 끊기지 않도록 방어하는 기능.
        *   **1.4.1.2. 텍스트 포맷팅 정규화 및 공백/특수문자 정제 기능:**  불필요하게 반복되는 연속 공백, 무의미한 줄바꿈(개행 문자), 깨진 인코딩 문자열 등을 처리하여 LLM과 임베딩 모델이 이해하기 쉬운 깔끔한 평문(Plain Text) 포맷으로 정규화하는 기능.
        *   **1.4.1.3. 의미 없는 단문(Short Text) 필터링 및 병합 기능:**  추출된 텍스트 조각 중 길이가 너무 짧거나(예: 1~2글자 특수기호) 단어로서의 의미를 갖지 못하는 찌꺼기 텍스트를 식별하여 분석 대상에서 제외하거나 앞뒤 문장에 병합하는 기능.
*   **1.5. 문서 구조 청킹(논리적 분할):**  정제된 텍스트를 임베딩 모델이 문맥을 소화하기 좋은 크기로 쪼개는 단계입니다.
    *   **1.5.1. 문서 구조 기반 논리적 분할:**  문서 내의 제목, 목차, 단락을 인식하여 문맥 단위로 1차 분할하는 기능.
        *   **1.5.1.1. 계층적 제목 기반 의미 단락(Section) 분할 기능:**  파싱된 텍스트의 스타일 구조를 바탕으로 큰 제목과 하위 내용을 하나의 의미 단락으로 묶어 1차 분할하는 기능.
        *   **1.5.1.2. 구조화 데이터(표, 목록) 무결성 보존 분할 기능:**  표나 목록 중간이 부자연스럽게 끊어지지 않도록, 해당 데이터 블록을 하나의 독립적인 단위로 취급하여 분할하는 기능.
    *   **1.5.2. 임베딩 최적화 크기 분할/청킹:**  1차로 나뉜 단락 중 너무 긴 구간을 임베딩 모델이 소화할 수 있는 크기(500~1000자 내외)로 한 번 더 나누는 기능.
        *   **1.5.2.1. 문맥 단절 방지를 위한 오버랩(Overlap) 분할 기능:**  1차 분할된 단락이 지정된 크기(500~1000자 내외)를 초과할 경우, 앞뒤 문장을 일정 부분 겹치게 자름으로써 문맥 단절을 방지하는 기능.
        *   **1.5.2.2. 청크별 컨텍스트 메타데이터(상위 제목 등) 병합 기능:**  최종 분할된 각 문서 조각(청크)에 해당 내용이 속했던 원본 파일명, 상위 절 제목, 페이지 번호 등의 문맥 정보를 태그 형태로 결합하여 저장하는 기능.
*   **1.6. 벡터 임베딩 및 데이터베이스 저장:**  분할된 문서 조각을 벡터로 변환해 검색 엔진이 빠르게 찾을 수 있도록 저장하는 단계입니다.
    *   **1.6.1. 청크 임베딩 생성 및 벡터 인덱싱:**  분할된 청크를 임베딩하여 초기 로컬 MVP용 FAISS 또는 향후 확장을 위한 AWS Bedrock Knowledge Base 등 유연한 벡터 데이터베이스에 인덱스로 저장하는 기능.
        *   **1.6.1.1. 텍스트 청크 다차원 벡터(Embedding) 변환 기능:**  문서 구조에 맞춰 분할을 마친 텍스트 조각(청크)들을 임베딩 모델(예: 외부 API 등)에 전달하여, 텍스트의 문맥적 의미를 담은 다차원 벡터 수치 데이터로 변환하는 기능.
        *   **1.6.1.2. 검색 식별용 메타데이터 결합 및 Vector DB (FAISS / Bedrock) 인덱스 저장 기능:**  변환 완료된 순수 벡터 데이터에 원본 파일명, 문서 내 순번(chunkId) 등 추후 출처를 추적할 수 있는 메타데이터를 결합하고, 이를 초기 로컬용 FAISS나 향후 클라우드용 AWS Bedrock Knowledge Base에 매핑하여 빠르고 정확한 탐색이 가능하도록 인덱싱하는 기능.
*   **1.7. 핵심 기능 및 요구사항 독립 추출 (RequirementExtractor):**  분할된 텍스트 내에서 시스템이 테스트해야 할 핵심 기능, 세부 요구사항, 사용자 흐름을 AI가 독립적인 단위로 식별해 내어 TDD 테스트 초안을 명확히 매핑할 수 있게 하는 기능.
    *   **1.7.1. 시스템 핵심 기능 및 사용자 흐름 식별 기능:**  문서의 전체 맥락을 분석하여 주요 기능(예: 파일 업로드, 로그인)과 API, 화면 단위의 사용자 작업 흐름을 큰 단위의 도메인 지식으로 묶어 식별하는 기능.
    *   **1.7.2. 검증 가능한 개별 요구사항 단위 분리 및 추출 기능:**  식별된 기능 내에서 '테스트 및 검증'이 가능한 구체적인 요구사항 문장(조건, 제약사항 등)만을 찾아내어 노이즈 없이 독립적인 단위로 분리해 내는 기능.
    *   **1.7.3. 추출 요구사항 고유 식별자(ID) 부여 및 메타데이터 기록 기능:**  추출 완료된 각 요구사항 항목에 고유 식별자(예: REQ-001)를 부여하고, 추후 생성될 테스트 케이스 초안과 1:N 근거 매핑이 명확하게 이루어질 수 있도록 H2 데이터베이스에 기본 정보(requirementText 등)를 기록하는 기능.
###### 2. 테스트 관점 및 TDD 초안 자동 생성
*   **2.1. 사용자 분석 관점 및 조건 설정:**  사용자 의도에 맞게 어떤 QA 관점으로 테스트를 생성할지 기준을 설정하는 단계입니다.
    *   **2.1.1. 명시적 QA 관점 선택 및 동적 추천:**  사용자가 QA 관점을 명시적으로 선택할 수 있는 인터페이스를 제공하고, 맞춤형 관점을 추가 추천하는 기능.
        *   **2.1.1.1. 사전 정의된 기본 QA 관점(백엔드, 프론트엔드, 보안 등) 선택 UI 제공 기능:**  사용자가 프론트엔드 대시보드에서 백엔드, 프론트엔드, API, 보안, 비기능 요구사항 등 시스템에 사전에 정의된 관심 QA 관점을 직접 선택할 수 있도록 체크박스 등의 인터페이스를 제공하는 기능.
        *   **2.1.1.2. 업로드 문서 문맥 분석 기반 맞춤형 QA 관점 동적 추천 기능:**  시스템이 업로드된 문서의 맥락을 사전 분석하여 해당 프로젝트에 가장 필요할 것으로 예상되는 '추천 QA 관점 리스트'를 도출하고 화면에 우선적으로 제안(렌더링)하는 기능.
    *   **2.1.2. 동적 관점 매개변수 조정 및 UI 반영:**  사용자의 선택(예: '보안', '성능')을 분석 프롬프트의 동적 변수로 주입하고, 실시간으로 반영된 매개변수와 프롬프트 구조를 사용자에게 보여주는 기능.
    *   **2.1.3. 프롬프트 엔지니어링 관점 자동 삽입:**  사용자가 별도의 지시를 하지 않아도 '구현 기반', '명세 기반', '설계 기반' 등 분석 관점을 프롬프트에 자동으로 주입하고 분석을 수행하는 기능.
    *   **2.1.4. 사용자 분석 조건 저장 및 재현성 확보:**  분석을 완료한 후 사용자가 설정한 관점과 매개변수 조건을 프로젝트별(AnalysisJob)로 저장하여, 동일한 조건으로 언제든 재분석할 수 있도록 지원하는 기능.
    *   **2.1.5. 사용자 지정 프롬프트 입력 기능:**  사용자가 LLM에게 특정 테스트 조건, 방향성, 추가 지시 사항을 전달할 수 있도록 프롬프트 입력창을 제공하는 기능.
*   **2.2. 프롬프트 생성 및 AI 분석 서버(LLM/RAG) 통신:**  검색된 근거와 사용자의 요청을 조합하여 지시를 내리고, 두 가지 형태의 분석 서버를 유연하게 교체하며 통신하는 단계입니다.
    *   **2.2.1. 검색 근거 및 사용자 프롬프트 기반 동적 조립 기능:**  RAG 문서 조각 또는 전체 텍스트를 생성 템플릿과 결합하고 사용자 지시사항을 반영하여 프롬프트를 구성하는 기능.
    *   **2.2.1.1. 시스템 지시문 및 구조화 데이터 주입 기능:**  LLM이 지시문과 데이터를 헷갈리지 않도록 명확한 식별 태그(<context> 등)로 분리 주입하는 기능.
    *   **2.2.1.2. LLM 컨텍스트 윈도우(Token) 초과 방지 최적화 기능:**  전송 텍스트가 최대 글자 수를 초과할 위험이 있을 때, 우선순위가 낮은 텍스트부터 잘라내어(Truncate) API 호출 실패를 사전에 방어하는 기능.
    *   **2.2.2. AI 분석 계층(Analysis Tier) 플러그 앤 플레이(Plug-and-Play) 교체 기능:**  완벽히 동일한 API 규격을 공유하는 두 가지 버전의 분석 서버를 백엔드 코드 수정 1줄 없이 URL 교체만으로 갈아 끼우는 기능.
        *   **2.2.2.1. LLM 서버 구동 모드 (MVP 핵심 기능 공장):**  복잡한 RAG(검색 증강) 과정 없이 추출된 텍스트 전체를 LLM에게 보내 "JSON 형태의 테스트 케이스를 짤 것"을 곧바로 지시(GENERATING)하는 최소 기능 모드. 초기 테스트 효율을 위해 가짜 응답을 반환하는 MockLlmClient와 임시 연동하여 구동할 수 있도록 지원.
        *   **2.2.2.2. RAG 서버 구동 모드 (최종 고도화 플랫폼):**  추출된 요구사항과 구축된 QA 지식베이스를 교차 검색한 뒤, 다중 근거가 결합된 복합 프롬프트를 조립하여 AWS Bedrock 등의 실제 외부 상용 LLM과 통신하는 고도화 모드.
*   **2.2.3. 비동기 작업 지시 및 단일 콜백(Webhook) 기반 무결성 통신:**
        *   **2.2.3.1. 비동기 작업 대기열(Queue) 위임 기능:**  백엔드가 직접 LLM 통신을 물고 있지 않고, 문서 번호(documentId)를 AI 분석 서버로 넘겨 분석을 비동기로 지시(QUEUED)하며, 프론트엔드는 대기 시간 동안 스피너 등 시각적 피드백을 렌더링하는 기능.
        *   **2.2.3.2. 웹훅(Webhook) 규격을 통한 구조화 결과 통합 출고 기능:**  분석 서버(LLM/RAG) 제조 과정이 무사히 끝나면, AI가 조립한 최종 JSON 결과물을 Spring Boot의 내부용 콜백 엔드포인트(/api/internal/analysis/{analysisId}/callback)로 쏘아주어 결과 매핑을 자동 완료(COMPLETED)시키는 결합도 낮은 통신 기능.
        *   **2.2.3.3. 통신 실패 시 사용자 친화적 예외 처리 (FAILED):**  분석 서버와의 통신 지연 또는 장애 발생 시 애플리케이션 전체 다운을 막고 상태를 실패로 변경한 후, 직관적인 에러 메시지를 반환하는 안전 종료 기능.
*   **2.3. 파싱 및 이력 데이터 관리:**  LLM의 텍스트 응답을 규격화된 QA 양식으로 변환하고 데이터베이스에 기록하는 단계입니다.
    *   **2.3.1. LLM 응답 데이터 구조화 파싱:**  텍스트 응답을 분석하여 사전 조건, 입력 데이터, 절차, 기대 결과 등 실제 QA 양식으로 쪼개는 기능.
        *   **2.3.1.1. 지정 포맷(JSON/Markdown 표) 식별 및 추출 기능:**  LLM이 반환한 자유 텍스트 응답 내용 속에서 시스템이 프롬프트로 지시했던 구조화된 데이터 포맷(예: JSON 블록 또는 Markdown 표) 영역만을 정확하게 찾아내어 텍스트 노이즈와 분리하는 기능.
        *   **2.3.1.2. QA 세부 항목 객체 매핑 기능:**  추출된 데이터를 분석하여 테스트 케이스 시나리오, 사전 조건, 입력 데이터, 테스트 절차, 기대 결과 등의 개별 필드로 명확히 쪼개고 데이터베이스에 저장할 수 있는 형태의 객체(Object)로 변환하는 기능.
        *   **2.3.1.3. 필수 데이터 누락 감지 및 예외(Fallback) 방어 기능:**  파싱 과정 중 LLM의 환각이나 형식 파괴로 인해 필수 항목(예: 기대 결과, 절차)이 누락되었음을 감지했을 때, 애플리케이션 중단을 막기 위해 '추가 검토 필요' 등의 기본값(Default)으로 채워 넣고 부분 성공으로 처리하는 안전장치 기능.
    *   **2.3.2. 생성 결과 단위 데이터베이스 저장:**  파싱된 개별 테스트 케이스를 생성 식별자와 함께 데이터베이스에 기록하는 기능.
    *   **2.3.3. 테스트 케이스 생성 이력 목록 및 단건 조회:**  한 번 생성된 테스트 케이스를 추후 조회할 수 있게 하는 기능.
*   **2.4. 테스트 우선순위 산정:**  생성된 항목 중 먼저 확인해야 할 핵심 케이스의 중요도를 평가하는 기능입니다.
    *   **2.4.1. 테스트 케이스 다면적 우선순위 자동 산정:**  요구사항 중요도 등을 고려하여 '높음/중간/낮음' 우선순위를 부여하는 기능.
        *   **2.4.1.1. 요구사항 중요도 및 사용자 영향도 분석 기능:**  생성된 테스트 케이스가 핵심 필수 기능과 연결되어 있는지, 그리고 실패했을 때 사용자 경험에 미치는 영향이 큰지를 분석하여 기초 중요도 점수를 산정하는 기능.
        *   **2.4.1.2. 오류 발생 가능성(예외/경계 조건) 가중치 부여 기능:**  정상 흐름 검증뿐만 아니라, 예외 상황 처리나 경계 조건이 포함되어 오류가 발생할 가능성이 높은 시나리오에 우선순위 가중치를 부여하는 기능.
        *   **2.4.1.3. 종합 등급(높음/중간/낮음) 할당 및 검토 요망 처리 기능:**  도출된 다면적 평가 지표(중요도, 오류 가능성, 근거 명확성 등)를 종합하여 최종적으로 '높음', '중간', '낮음' 또는 '추가 검토 필요' 중 하나의 등급으로 확정하고 결과 데이터베이스에 기록하는 기능.
*   **2.5. 위험 요소 탐지:**  문서 문맥을 바탕으로 개발/운영 시 예상되는 결함 유형을 찾아내는 기능입니다.
    *   **2.5.1. 문맥 기반 동적 위험 요소 식별 및 맞춤형 태그 부착:**  결함 발생 가능 구간을 분석하여 가장 적합한 위험 유형 태그를 생성해 부착하는 기능.
        *   **2.5.1.1. 요구사항 문맥 기반 핵심 결함(위험) 유형 식별 기능:**  추출된 요구사항을 분석하여 '인증 실패', '입력값 오류', '통신 실패' 등 QA 관점에서 발생 가능성이 높은 기능적/보안적 위험 요소를 동적으로 도출하는 기능.
        *   **2.5.1.2. 식별된 위험 요소의 해시태그(Tag) 변환 및 컴포넌트 부착 기능:**  도출된 위험 요소들을 단순 텍스트로 나열하지 않고, 사용자가 직관적으로 인지할 수 있도록 해시태그 형태의 UI로 변환하여 생성된 테스트 케이스 카드 내에 부착(렌더링)하는 기능.
###### 3. 위험 요소 탐지 및 근거 매핑
*   **3.1. 문서 근거 검색:**  벡터 DB에서 테스트 생성의 기반이 될 가장 연관성 높은 원문 단락을 찾아오는 단계입니다.
    *   **3.1.1. 벡터 유사도 기반 관련 문서 청크 검색:**  벡터 DB를 탐색해 현재 요청과 연관성이 높은 업로드 문서 조각을 찾아오는 기능.
        *   **3.1.1.1. 검색 질의(Query) 문맥 임베딩 변환 기능:**  사용자가 입력한 요구사항 텍스트와 선택한 QA 관점을 조합한 질의 문자열을 생성하고, 이를 임베딩 모델을 통해 벡터 데이터로 변환하는 기능.
        *   **3.1.1.2. Vector DB 기반 Top-K 유사도 탐색 기능:**  변환된 질의 벡터를 활용하여 FAISS 또는 AWS Bedrock Knowledge Base 내에서 코사인 유사도가 가장 높은 상위 K개의 관련 문서 조각(청크)을 빠른 속도로 추출해 내는 기능.
        *   **3.1.1.3. 검색 결과 품질 제어(Threshold 필터링) 기능:**  추출된 문서 조각들 중 유사도 점수가 시스템이 정한 최소 기준치(Threshold) 미만인 의미 없는 노이즈 데이터를 차단하고 필터링하여 RAG의 정확도를 높이는 기능.
    *   **3.1.2. 사전 구축된 'QA 지식베이스' 연동 및 복합 검색:**  업로드 문서뿐만 아니라 시스템 내부에 미리 정제되어 저장된 QA 지식베이스(테스트 설계 기준, 보안 체크리스트, 웹/API 위험 사례)를 연동해 복합적으로 검색하고 위험 요소 탐지에 활용하는 기능.
        *   **3.1.2.1. QA 지식베이스 전용 인덱스(Vector Space) 연동 기능:**  사용자 업로드 문서와 섞이지 않도록, 사전에 정제된 QA 도메인 지식(테스트 레벨, 보안 체크리스트 등)만을 별도의 독립적인 벡터 데이터베이스(초기 로컬 FAISS 및 향후 AWS Bedrock Knowledge Base 지원) 공간으로 구축하여 연동하는 기능.
        *   **3.1.2.2. 요구사항 문맥 기반 QA 지식 교차 검색(Cross-Search) 기능:**  사용자가 업로드한 요구사항 문장에서 추출된 핵심 키워드나 식별된 위험 요소를 질의(Query)로 삼아, 연동된 QA 지식베이스 내에서 해당 기능에 가장 적합한 테스트 관점이나 과거 결함 사례를 교차 탐색하는 기능.
        *   **3.1.2.3. 사용자 문서 및 지식베이스 검색 결과 가중치 병합 기능:**  사용자의 원본 문서에서 찾은 근거(명세 기반 규칙)와 QA 지식베이스에서 찾은 근거(테스트 설계 기준)를 하나의 컨텍스트로 취합하고, LLM 프롬프트에 주입하기 위해 관련성 점수에 따라 최종 순위를 매겨 병합하는 기능.
*   **3.2. 메타데이터 추출 및 N:1 다중 매핑:**  찾은 문서 조각의 정보를 분석하여 여러 개의 문서 근거를 하나의 테스트 케이스에 매핑하는 단계입니다.
    *   **3.2.1. 검색 근거 메타데이터 구성 및 유사도 평가:**  검색된 문서 조각에서 원본 파일명, 청크 순번, 유사도 점수 등의 메타데이터를 추출하고 평가하는 기능.
    *   **3.2.2. 테스트 케이스-다중 근거(1:N) 매핑 및 DB 저장:**  추출된 정보를 활용해 하나의 테스트 케이스에 최대 3개의 유효 근거를 연결하고 저장하는 기능.
        *   **3.2.2.1. 최적 근거 선별 및 참조 개수 제한(최대 3개) 기능:**  벡터 검색을 통해 도출된 관련 문서 조각들 중에서 유사도 점수가 높은 핵심 근거만을 필터링하고, 너무 많은 텍스트가 혼입되는 것을 막기 위해 참조 개수를 최대 3개로 제한하는 기능.
        *   **3.2.2.2. 요구사항-테스트-근거 1:N 추적성(Traceability) 객체 결합 기능:**  단일 테스트 케이스(1) 객체 내부에 여러 개의 문서 근거 식별자(N)가 논리적으로 포함되도록 데이터 구조를 결합하여, 테스트와 근거 간의 직접적인 추적성을 확보하는 기능.
*   **3.2.2.3. 매핑 데이터베이스(DB) 트랜잭션 기록 기능:**  구조화가 완료된 1:N 매핑 데이터(테스트 케이스 결과 및 근거 메타데이터)를 H2 등의 데이터베이스에 매핑 테이블 형태로 정확히 저장하여 누락을 방지하는 기능.
*   **3.3. 매핑 무결성 방어:**  잘못 참조된 인덱스로 시스템이 중단되는 것을 막는 자동 보호 단계입니다.
    *   **3.3.1. 테스트 케이스 근거 인덱스 검증 및 Fallback 처리:**  LLM 반환 인덱스가 유효 범위를 벗어나면 근거 없는 결과로 간주하여 처리하는 기능.
        *   **3.3.1.1. LLM 응답 근거 인덱스 유효성(Out-of-Bounds) 검사 기능:**  LLM이 파싱 응답으로 반환한 근거 식별자(ID)나 인덱스가 실제 시스템이 프롬프트로 제공했던 유효 문서 조각(청크) 리스트 내에 정상적으로 존재하는지 백엔드에서 1차로 무결성을 검증하는 기능.
        *   **3.3.1.2. 무효 인덱스 탐지 시 결과 제외 및 신뢰성 확보(Fallback) 기능 (수정됨):**  잘못된 인덱스(환각)로 판별될 경우 애플리케이션 에러를 막기 위해 임의의 문서 조각(Top-1 청크)에 강제로 매핑하지 않고, '근거 없는 결과 제한' 원칙에 따라 해당 테스트 케이스를 결과에서 기본적으로 제외하거나, 사용자 화면에 '근거 부족(추가 검토 필요)' 상태로 명확히 표시하여 시스템의 신뢰성을 유지하는 기능.
        *   **3.3.1.3. Fallback 처리 이력 로깅(Logging) 및 환각 빈도 추적 기능:**  무효 인덱스 처리(제외 또는 경고 표시)가 발생했을 때 이를 조용히 넘기지 않고 서버 시스템 로그(Log)로 상세 내역을 기록하여, 추후 프롬프트 엔지니어링을 개선하거나 LLM의 인덱스 환각 발생 빈도를 개발자가 모니터링할 수 있도록 하는 기능.
*   **3.4. 근거 데이터 시각적 UI 제공:**  사용자 화면에서 복잡한 근거 텍스트를 직관적으로 제어하고 표현하는 단계입니다.
    *   **3.4.1. 근거 부족 테스트 케이스 시각적 경고 렌더링:**  연결된 근거가 없거나 기준치 미달 시 시각적인 경고 마크를 띄우는 기능.
    *   **3.4.2. 장문 근거 데이터의 '접기/펼치기' 및 축약 UI 제공:**  긴 근거 문장을 축약하고 아코디언 UI 등 확장 제어 버튼을 제공하는 기능.
        *   **3.4.2.1. 텍스트 길이 감지 및 말줄임표(Truncation) 자동 적용 기능:**  렌더링할 근거 데이터의 텍스트가 지정된 길이(예: 300자)나 줄 수를 초과할 경우, 카드 UI가 무너지지 않도록 첫 2~3줄만 화면에 그리고 끝부분을 말줄임표(...)로 자동 축약 처리하는 렌더링 기능.
        *   **3.4.2.2. 아코디언(Accordion) 방식 원문 확장/축소(Toggle) 상호작용 기능:**  축약된 텍스트 하단에 '더 보기(접기/펼치기)' 버튼을 제공하여, 사용자가 클릭할 때마다 숨겨져 있던 원문 전체가 부드럽게 펼쳐지거나 다시 접히는 시각적 동적 상호작용(Interaction)을 제어하는 기능.
###### 4. TDD 검증 보고서 산출
*   **4.1. 시각적 진행 상태 및 결과 모니터링:**  시스템 내부에서 일어나는 전 과정과 최종 결과를 사용자 친화적인 대시보드 화면에 보여주는 단계입니다.
    *   **4.1.1. 파이프라인 단계별 진행 상태 및 통계 모니터링 UI:**  전처리, 임베딩 등 전 과정의 진행률과 상태를 실시간 대시보드에 제공하는 기능.
    *   **4.1.2. 분석 결과 단계별 카드형 시각화:**  도출된 요약, 테스트 케이스 상세 시나리오 등을 직관적인 카드 형태로 렌더링하는 기능.
        *   **4.1.2.1. 요구사항 명세서 전체 요약 카드 렌더링 기능:**  문서의 목적, 주요 기능, 테스트 대상 범위 등을 비전공자도 이해하기 쉬운 설명형 문장으로 요약하여 대시보드 최상단에 독립된 카드로 표시하는 기능.
        *   **4.1.2.2. 구조화된 테스트 케이스 상세(절차, 조건 등) 컴포넌트 렌더링 기능:**  생성된 테스트 케이스의 단순 제목뿐만 아니라 사전조건, 입력 데이터, 테스트 절차, 기대 결과 등의 세부 항목들을 사용자가 읽기 쉽게 분할된 카드 UI로 화면에 그리는 기능.
        *   **4.1.2.3. 우선순위 배지 및 위험 요소 태그 시각적 강조 기능:**  각 테스트 케이스별 중요도(높음/중간/낮음)를 한눈에 파악할 수 있도록 색상 배지로 나타내고, 발생 가능한 위험 요소(예: 인증 실패)를 해시태그 형태로 카드 내에 부착하여 주의를 환기하는 기능.
*   **4.2. 컨텍스트 파이프라인 제어:**  사용자가 헤매지 않도록 시스템 맥락에 맞는 다음 추천 동작을 띄워주는 기능입니다.
    *   **4.2.1. 컨텍스트 인식형 원클릭 파이프라인 제어기:**  현재 문서 상태에 따라 다음 추천 작업 버튼을 제공해 한 번의 클릭으로 백엔드 동작을 연결하는 기능.
*   **4.3. 결과 종합, 파일 변환 및 메타데이터 관리:**  개별 생성 데이터를 취합하여 최종 포맷의 문서로 변환하고 데이터를 저장·관리하는 단계입니다.
    *   **4.3.1. 보고서 도메인 데이터 종합 및 구조화:**  요약, 테스트 목록, 근거 등 생성 데이터를 모아 하나의 보고서 구조로 취합하는 기능.
        *   **4.3.1.1. 프로젝트 개요 및 분석 명세서 요약 정보 취합 기능:**  보고서 서두 구성을 위해 사용자가 초기에 입력한 프로젝트 기본 정보(이름, 개요 등)와 시스템이 생성한 요구사항 명세서 전체 요약 데이터를 DB에서 조회하여 취합하는 기능.
        *   **4.3.1.2. 테스트 케이스 시나리오 및 1:N 다중 근거 병합 기능:**  본문에 들어갈 테스트 케이스 상세 항목(절차, 기대 결과 등)과 각 케이스에 연결된 최대 3개의 유효 근거 데이터를 논리적인 계층 구조로 묶어 병합하는 기능.
        *   **4.3.1.3. 최종 보고서 데이터 모델(DTO) 조립 및 무결성 검증 기능:**  취합된 모든 데이터를 문서 변환 모듈이 요구하는 최종 보고서 템플릿 양식에 맞추어 하나의 데이터 객체로 조립하고, 문서 생성 전 필수 값 누락 여부를 검증하는 기능.
    *   **4.3.2. Markdown 및 PDF 포맷 문서 렌더링:**  취합된 종합 데이터를 바탕으로 Markdown 및 PDF 양식에 맞추어 변환하는 기능.
        *   **4.3.2.1. Markdown 검증 보고서 생성:**  도출된 종합 데이터를 정해진 템플릿에 맞추어 마크다운 문서로 조립하는 기능.
        *   **4.3.2.2. PDF 검증 보고서 렌더링 및 변환:**  Markdown 데이터를 기반으로 시각적인 스타일을 적용해 PDF 포맷으로 렌더링하는 기능.
    *   **4.3.3. 보고서 파일 물리적 저장(S3) 및 유실 복구:**  최종 파일을 S3 등에 저장하고 분실 시 DB 정보로 자동 재생성하는 보호 기능.
*   **4.3.3.1. 생성 완료 보고서의 물리적(로컬/S3) 저장 기능:**  사용자가 검토를 마치고 보고서 생성을 확정했을 때, 생성된 최종 파일(Markdown 등)을 S3(초기 로컬 환경에서는 MinIO 등 S3 호환 스토리지 사용)에 안전하게 기록하는 기능.
        *   **4.3.3.2. 다운로드 요청 시 물리적 파일 존재 여부(무결성) 검증 기능:**  사용자가 파일 다운로드를 요청하는 시점에, 해당 경로에 실제 파일이 남아있는지 확인하고 유실 여부를 즉각적으로 판단하는 검증 기능.
        *   **4.3.3.3. DB 메타데이터 기반 백그라운드 자동 재생성(무중단 복구) 기능:**  서버 환경 문제 등으로 로컬 파일 유실이 감지되었을 때, 사용자에게 에러를 띄우지 않고 DB에 저장된 도메인 정보를 바탕으로 백그라운드에서 문서를 즉시 자동 재생성하여 정상적인 다운로드 흐름을 이어가는 기능.
    *   **4.3.4. 보고서 메타데이터 DB 저장 및 다중 조건 조회/삭제:**  생성 이력을 DB로 관리하고 특정 기준으로 목록을 조회/삭제하는 기능.
        *   **4.3.4.1. 보고서 생성 이력 및 메타데이터 안전 기록 기능:**  최종 파일이 물리적으로 생성되었을 때, 해당 파일의 저장 경로, 생성 일시, 파일 형식 등의 메타데이터를 데이터베이스(예: H2 DB)에 매핑하여 누락 없이 기록하는 기능.
        *   **4.3.4.2. 원본 문서 및 생성 이력 기반 다중 필터링 조회 기능:**  대시보드 화면에서 사용자가 특정 원본 문서(documentId)나 분석 작업 단위의 생성 이력(generationId)을 기준으로 기존에 생성된 보고서 목록을 필터링하여 빠르고 정확하게 찾아볼 수 있도록 제공하는 기능.
        *   **4.3.4.3. 보고서 단건 삭제 및 물리적 파일 동기화 파기 기능:**  사용자가 대시보드에서 특정 보고서를 삭제 요청했을 때, DB 내의 메타데이터만 지우는 것이 아니라 실제 S3(또는 S3 호환 스토리지)에 보관된 마크다운/PDF 파일 원본까지 찾아내어 연쇄적으로 완전 파기하는 기능.
    *   **4.3.5. '보완 필요 사항' 및 '한계/주의사항' 자동 도출 및 병합:**  최종 검증 보고서 산출 시 생성된 테스트 케이스 외에 프로젝트 상 누락되거나 보완이 필요한 영역, AI 생성 결과물의 한계점을 자동 도출하여 최종 보고서에 병합하는 기능.
        *   **4.3.5.1. 명세서 대비 누락 및 보완 필요 사항 동적 추론 기능:**  추출된 요구사항과 QA 지식베이스를 교차 검증하여, 예외 처리 누락이나 비기능적 측면(보안 등)에서 추가로 정의되어야 할 테스트 관점을 AI가 식별해 내는 기능.
        *   **4.3.5.2. 생성 결과 한계 고지 및 수동 검토 유도(Disclaimer) 템플릿 삽입 기능:**  "본 시스템은 테스트 수행 결과를 보장하거나 최종 판단을 대체하지 않으며, QA 담당자의 검토를 전제로 합니다"라는 시스템의 필수 안내 문구 및 한계점을 보고서 양식에 고정으로 삽입하는 기능.
        *   **4.3.5.3. 보고서 하단 독립 목차 할당 및 통합 렌더링 기능:**  도출된 보완 필요 사항과 고정된 한계/주의사항 데이터를 QA 보고서 템플릿의 후반부(예: 목차 8번, 9번)에 독립된 섹션으로 할당하여 다른 분석 데이터와 충돌 없이 병합 렌더링하는 기능.
*   **4.4. 브라우저 기반 보고서 렌더링:**  별도의 리더기 없이 시스템 내에서 최종 보고서를 점검하는 기능입니다.
    *   **4.4.1. 브라우저 기반 보고서 렌더링 및 미리보기 UI:**  다운로드 전 화면 내에서 보고서 전체 내용을 읽고 검토하는 미리보기 기능.
*   **4.5. 보고서 클라이언트 반출:**  확정된 파일을 외부 기기로 빼내는 기능입니다.
    *   **4.5.1. 최종 보고서 파일 다운로드 연동:**  최종 생성된 문서를 클라이언트로 다운로드 받을 수 있게 연결하는 기능.
